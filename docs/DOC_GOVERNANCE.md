# 文档治理规则（Document Governance）

## 一、适用范围
本文档定义本项目中所有文档的治理细则，用于指导：
- 多 Codex 并行完善文档
- 文档成熟度评估
- 文档进展更新
- 文档责任边界管理
- 需求变更回写
- 共享契约冲突处理
- 子任务双文档治理
- 多轮工作收口与合并

本文档适用于：
- 总控 Codex
- 模块 Codex
- 子任务 Codex
- 评审 Codex

## 二、文档分层与职责

### 1. 全局层
用于维护全项目统一视角。

#### 文档
- `ORIGINAL_REQUIREMENTS.md`
- `ORIGINAL_PLAN.md`
- `PLAN_LATEST.md`
- `TASK_INDEX.md`
- `EXECUTION_LOG.md`
- `TECHNICAL_STANDARDS.md`
- `DESIGN_DECISIONS.md`
- `MODULE_INDEX.md`
- `OPEN_QUESTIONS.md`
- `DOCUMENT_MATURITY.md`
- `DOCUMENT_PROGRESS.md`

#### 职责
- 提供项目整体背景、计划、索引、规范、成熟度、进展和全局问题视图
- 不承载模块内部细节和子任务级实施细节

---

### 2. 模块层
用于承接模块级需求与设计。

#### 文档
- `MODULE_REQUIREMENTS.md`
- `MODULE_DESIGN.md`
- `MODULE_API_DESIGN.md`
- `MODULE_SCHEMA_DESIGN.md`
- `MODULE_LOGIC_DESIGN.md`
- `MODULE_TASK_INDEX.md`
- `MODULE_DEPENDENCIES.md`
- `MODULE_EXECUTION_LOG.md`
- `MODULE_OPEN_QUESTIONS.md`

#### 职责
- 承接全局需求，细化为模块需求
- 承接模块设计、模块接口、模块数据、模块逻辑
- 建立模块内部任务索引、依赖关系和模块级问题跟踪

---

### 3. 子任务层
用于承接单个子任务的设计与实施。

#### 文档
- `SUBTASK_DESIGN.md`
- `SUBTASK_IMPLEMENTATION.md`

#### 职责
- `SUBTASK_DESIGN.md`：定义该子任务如何被设计
- `SUBTASK_IMPLEMENTATION.md`：定义该子任务如何被实施

#### 禁止事项
- 不允许在子任务文档中重新定义模块级共享契约
- 不允许把上游设计缺口直接隐藏到实施步骤里
- 不允许混淆设计和实施职责

## 三、角色责任边界

### 1. 总控 Codex
#### 默认负责
- `PLAN_LATEST.md`
- `TASK_INDEX.md`
- `DOCUMENT_MATURITY.md`
- `DOCUMENT_PROGRESS.md`
- `MODULE_INDEX.md`
- `OPEN_QUESTIONS.md`
- `TECHNICAL_STANDARDS.md`
- `DESIGN_DECISIONS.md`

#### 责任
- 统一维护文档治理状态
- 识别低成熟度关键文档
- 识别适合并行的工作单元
- 统一更新成熟度与进展
- 合并多轮并行结果
- 统一维护全局待确认问题
- 决定何时允许开启子任务级并行

---

### 2. 模块 Codex
#### 默认负责
当前被分配模块下的所有模块级文档。

#### 责任
- 完善模块需求、设计、API、schema、logic、任务索引、依赖关系
- 推动模块级文档达到下游可用状态
- 显式记录模块级待确认问题和阻塞
- 不能修改全局总控文档
- 不能修改其他模块文档

---

### 3. 子任务 Codex
#### 默认负责
当前被分配子任务目录下的：
- `SUBTASK_DESIGN.md`
- `SUBTASK_IMPLEMENTATION.md`

#### 责任
- 先检查上游成熟度
- 优先完善子任务设计文档
- 设计足够成熟后再完善实施文档
- 保持设计与实施职责分离
- 不能擅自修补模块级契约缺口

---

### 4. 评审 Codex
#### 责任
- 检查多个 Codex 产出是否一致
- 检查成熟度标注是否合理
- 检查共享契约是否漂移
- 检查子任务双文档是否职责分离
- 检查模块间 API / schema / logic 风格是否一致

## 四、多 Codex 并行规则

### 1. 可并行的范围
#### 模块级并行
不同模块可并行完善，但必须满足：
- 各模块边界清晰
- 全局规范足够稳定
- 没有多个 Codex 同时修改同一模块文档

#### 子任务级并行
不同子任务可并行完善，但必须满足：
- 所属模块文档已达到可作为下游输入的成熟度
- 子任务不重定义共享契约
- 子任务之间没有隐式耦合未解

---

### 2. 不适合并行的范围
以下文档通常不应由多个 Codex 同时自由修改：
- `PLAN_LATEST.md`
- `TASK_INDEX.md`
- `DOCUMENT_MATURITY.md`
- `DOCUMENT_PROGRESS.md`
- `MODULE_INDEX.md`
- `OPEN_QUESTIONS.md`
- `TECHNICAL_STANDARDS.md`
- `DESIGN_DECISIONS.md`

这些文档默认由总控 Codex 维护。

---

### 3. 共享契约锁定规则
如果某份文档是以下内容的事实来源：
- API 契约
- 数据结构契约
- 逻辑约束
- 错误码规范
- 状态机规范
- 跨模块依赖关系

则该文档应视为共享契约文档。

共享契约文档规则：
1. 默认不允许多个 Codex 同时自由修改
2. 若需修改，必须由主责代理或总控代理主导
3. 若修改会影响下游子任务，必须先暂停相关并行工作并记录影响

## 五、文档成熟度模型

### 1. 统一成熟度等级
- `L0` 未创建
- `L1` 仅有骨架
- `L2` 初稿
- `L3` 待澄清
- `L4` 可评审
- `L5` 可作为下游输入
- `L6` 可直接用于实施
- `L7` 稳定维护中

### 2. 各等级判定说明
#### L0 未创建
文档不存在，或只有文件名没有实质内容。

#### L1 仅有骨架
文档结构和章节已经建立，但内容几乎为空。

#### L2 初稿
已有主要内容，但缺失大量关键信息，不能作为稳定依据。

#### L3 待澄清
已有较多内容，但关键歧义仍多，不能安全作为下游依据。

#### L4 可评审
内容基本完整，可由人工或评审 Codex 审阅，但仍不适合直接实施。

#### L5 可作为下游输入
内容足够稳定，可支撑下游文档继续建设。

#### L6 可直接用于实施
内容已足够明确，可支撑实施执行。

#### L7 稳定维护中
文档已脱离草稿阶段，进入持续维护状态。

### 3. 典型目标
- 模块级文档通常至少需要达到 `L5`，才能支撑子任务设计
- 子任务设计文档通常至少达到 `L5`，才能支撑子任务实施文档
- 子任务实施文档通常至少达到 `L6`，才能进入实施执行

## 六、文档进展更新规则

### 1. `DOCUMENT_PROGRESS.md` 的职责
用于记录整套文档体系当前进展，包括：
- 当前阶段
- 当前已完成文档类型
- 当前已成熟的模块
- 当前已具备实施条件的子任务
- 当前阻塞项
- 当前待确认问题
- 当前下一优先工作
- 当前整体完成度评估

### 2. 每轮必须考虑更新的状态文档
每轮工作结束后，都必须检查是否需要更新：
- `DOCUMENT_MATURITY.md`
- `DOCUMENT_PROGRESS.md`
- `OPEN_QUESTIONS.md`

必要时还应更新：
- `TASK_INDEX.md`
- `PLAN_LATEST.md`
- `MODULE_INDEX.md`
- 模块级 `MODULE_EXECUTION_LOG.md`
- 模块级 `MODULE_OPEN_QUESTIONS.md`

### 3. 每轮收口至少要输出
1. 本轮新增了哪些文档
2. 本轮更新了哪些文档
3. 哪些文档成熟度有提升
4. 当前还缺哪些关键文档
5. 当前新增了哪些待确认问题
6. 当前下一轮建议优先补什么

## 七、需求变更回写规则

### 1. 基本原则
任何需求补充、需求澄清、范围变更，都不能只停留在某一个 Codex 的局部上下文中。

### 2. 处理步骤
#### Step 1：识别变更类型
判断属于：
- 需求澄清
- 范围变更
- 验收标准变更
- 技术约束补充
- 新增模块需求
- 新增子任务需求
- 共享契约变更

#### Step 2：判断影响范围
判断影响：
- 全局
- 某个模块
- 某个子任务
- 多个模块
- 多个子任务

#### Step 3：同步回写
根据影响范围更新：
- `OPEN_QUESTIONS.md`
- `DOCUMENT_MATURITY.md`
- `DOCUMENT_PROGRESS.md`
- `PLAN_LATEST.md`
- `TASK_INDEX.md`
- 对应模块文档
- 对应子任务文档

#### Step 4：必要时暂停并行
如果变化影响共享契约或多个下游任务：
- 暂停相关并行工作
- 标记阻塞
- 输出影响分析
- 由总控 Codex 统一重新分配

## 九、子任务可实施判定规则
一个子任务只有在以下条件都满足时，才应被视为“可实施”：

1. 模块需求文档至少达到 `L5`
2. 模块设计文档至少达到 `L5`
3. 模块 API / schema / logic 文档至少达到 `L5`
4. `SUBTASK_DESIGN.md` 至少达到 `L5`
5. `SUBTASK_IMPLEMENTATION.md` 至少达到 `L6`
6. 不存在未处理的关键共享契约冲突
7. 不存在阻塞性的待确认问题

若不满足上述条件，必须：
- 明确标记缺失项
- 记录阻塞原因
- 记录建议先补哪些上游文档

## 十、阻塞与待确认问题处理规则

### 1. 什么情况属于阻塞
以下情况通常应视为阻塞：
- 上游文档成熟度不足
- 共享契约缺失
- 模块边界不清
- 子任务边界不清
- 需求存在明显歧义
- 设计存在多条互斥路径但未确认
- 多个模块间依赖未定义
- 当前文档与其他文档冲突

### 2. 遇到阻塞时必须做什么
1. 不要自行发明关键契约
2. 记录阻塞原因
3. 记录影响范围
4. 写入相应待确认问题文档
5. 向总控 Codex 回报
6. 推荐下一步应优先补的文档

### 3. 待确认问题最少应包含
- 编号
- 问题描述
- 影响范围
- 当前阻塞的工作
- 建议方案
- 建议由谁确认

## 十一、收口与合并规则

### 1. 模块轮次收口
模块 Codex 每轮结束后至少应回报：
1. 更新了哪些文档
2. 每份文档当前建议成熟度
3. 本轮新增的待确认问题
4. 当前模块还缺什么
5. 哪些子任务已可继续推进
6. 哪些子任务仍被阻塞

### 2. 子任务轮次收口
子任务 Codex 每轮结束后至少应回报：
1. `SUBTASK_DESIGN.md` 当前成熟度建议
2. `SUBTASK_IMPLEMENTATION.md` 当前成熟度建议
3. 当前关键缺失项
4. 当前阻塞项
5. 当前待确认问题
6. 当前是否已具备实施条件

### 3. 总控收口
总控 Codex 每轮结束后应：
1. 合并成熟度变化
2. 合并进展变化
3. 合并待确认问题
4. 识别新的并行机会
5. 判断是否允许开启更多子任务级工作
6. 更新下一轮建议优先项

## 十二、推荐工作节奏

### 阶段 1：全局骨架稳定
目标：
- 全局规则稳定
- 全局索引稳定
- 成熟度和进展文档建立

### 阶段 2：模块级文档推进到 L5
目标：
- 模块需求 / 设计 / API / schema / logic 足够稳定
- 模块内部任务索引基本成型

### 阶段 3：子任务设计推进到 L5
目标：
- 关键子任务已有稳定设计依据

### 阶段 4：子任务实施文档推进到 L6
目标：
- 关键子任务具备执行条件

### 阶段 5：执行与持续维护
目标：
- 文档体系进入增量演进状态
- 成熟度与进展持续更新
## 十三、文档模板

### SUBTASK_DESIGN.md文档定义及规则

#### 作用

这份文档用于定义：
- 这个子任务到底要解决什么问题
- 它依赖哪些上游模块设计
- 它的输入输出是什么
- 它用到哪些接口契约、数据结构和逻辑约束
- 它的正确实现应满足什么条件
- 它有哪些风险、边界和待确认问题

它不是执行步骤清单。
它的目标是让实施前，先把“设计依据”说清楚。

#### 放到哪里

也放在同一个子任务目录下：docs/modules/<module>/sub_modules/<subtask>/SUBTASK_DESIGN.md

例如：docs/modules/M02-identity-and-team/sub_modules/ST02_03-authorization-matrix/SUBTASK_DESIGN.md

#### 谁来写

主要由子模块 / 子任务 Codex写。
但前提是上游模块文档已经足够成熟。

### SUBTASK_IMPLEMENTATION.md文档定义及规则
#### 作用

这份文档用于定义：
- 这个子任务具体怎么实施
- 允许改哪些文件
- 不能碰哪些区域
- 实施步骤是什么
- 每步怎么验证
- 完成定义（DoD）是什么
- 回滚怎么做
- 执行后要更新哪些文档
它不是设计文档。
它的目标是让实际执行的人或 Codex 可以按步骤做，不需要临场发明实施方式。

#### 放到哪里

也放在同一个子任务目录下：docs/modules/<module>/sub_modules/<subtask>/SUBTASK_IMPLEMENTATION.md

例如：docs/modules/M02-identity-and-team/sub_modules/ST02_03-authorization-matrix/SUBTASK_IMPLEMENTATION.md

#### 谁来写

也是主要由子模块 / 子任务 Codex写。
