# R1 Penpot 低保真页面规格

## 1. 文档定位

本文档用于 R1 acceptance closeout 前的页面设计校准，目标是把现有 R1 需求、设计、前端实现、E2E 断言和用户反馈整理为 Penpot 低保真页面输入。

本文档只定义低保真页面规格、Penpot frame 规划、Penpot MCP 后续执行方式和后续窗口计划。

本文档不修改业务代码，不修改前端代码，不修改测试，不修改数据库，不修改 `docs/governance/**`，不修改 `DOC_STATE.yaml`，不生成 implementation packet，不进入 R1 closeout，不声明任何任务 implementation-ready。

## 3. 全局页面原则

### 3.1 布局优先级

后续前端实现的布局优先级固定为：

1. 已经由用户确认的 Penpot frame。
2. 本文档 low-fi spec。
3. 现有需求和设计文档。
4. 当前前端代码和 E2E 断言。

现有代码只能作为能力事实和字段事实，不能作为最终布局依据。

### 3.2 文案分层规则

低保真页面稿必须区分三层语言，避免把工程、测试或状态机术语直接放到用户可见界面上。

| 语言层 | 使用位置 | 表达要求 |
| --- | --- | --- |
| 用户可见语言 | Penpot frame、页面标题、区块标题、按钮、状态提示、前端主文案 | 使用业务语言，例如“工作台概览”“最近模拟记录”“复盘结果”“参考资料”“证据不足”“评分理由”“改进建议”“薄弱项”“可重新导出”“暂无记录”“加载失败”“部分资料不可用”“数据已保存，可随时回看”。 |
| 产品说明语言 | 本文档解释段落、页面目标、低保真意图说明 | 可以说明“该区域用于表达数据可恢复能力”“该区域用于展示复盘证据链”“该状态用于提示资料不足导致可信度下降”。 |
| 工程验收语言 | Codex 实现注意事项、验收规则、禁止项、开发文档引用 | 可以保留 `PostgreSQL`、`traceability`、`E2E`、`RAG citation`、`degraded`、`failed`、`empty`、`legacy mock`、`hidden resource id` 等内部术语，但不得作为页面主文案或 frame 画布正文。 |

页面稿中出现技术状态时，必须先转换为用户能理解的业务状态；工程字段名只允许出现在实现说明、验收说明或代码相关引用中。

### 3.3 全局 App Shell / 信息架构

R1 所有主页面必须采用统一 App Shell。App Shell 是后续 Penpot frame 和前端实现的最高优先级布局约束，优先级高于当前前端代码中已有的顶部导航形态。

统一 App Shell 必须包含：

| 区域 | 低保真要求 |
| --- | --- |
| 左侧全局导航 | 固定承载 R1 主要业务栏目，导航选中态随右侧详情区路由变化。 |
| 右侧页面详情区 | 承载当前页面标题、主内容、状态区、主要操作和空态 / 失败态 / 降级态。 |
| 右上角用户登录信息区 | 在右侧详情区顶部或全局顶栏右侧展示用户头像、用户名称、个人配置、偏好设置和退出登录入口。 |

左侧导航只承载主要业务栏目，不放用户个人配置。用户个人配置必须放在右上角头像下拉中。旧 mock 入口不得出现在主导航，只允许作为开发辅助入口或隐藏入口。

左侧全局导航推荐 IA：

| 一级栏目 | 二级入口 |
| --- | --- |
| 工作台总览 | 工作台首页 |
| 简历和岗位管理 | 岗位管理、简历管理 |
| 模拟面试 | 发起模拟面试、模拟面试历史 |
| 复盘 | 复盘列表、评分 / 复盘详情 |
| 知识库 | 我的资料、公共知识库、参考资料状态 |

右上角用户登录信息区至少包含：

- 用户头像。
- 用户名称或账号标识。
- 下拉菜单。
- 个人配置入口。
- 偏好设置入口。
- 退出登录入口。

用户下拉菜单建议包含：

- 个人配置。
- 偏好设置。
- 安全与隐私。
- 退出登录。

右侧详情区按页面类型切换：

| 页面类型 | 右侧详情区内容 |
| --- | --- |
| 工作台首页 | 工作台概览、快捷入口、最近记录、复盘能力摘要。 |
| 简历和岗位管理 | 列表、创建入口、状态、空态。 |
| 模拟面试 | 发起流程、历史记录、面试详情。 |
| 复盘 | 复盘列表、评分详情、证据链、建议、薄弱项。 |
| 知识库 | 资料列表、资料处理状态、参考资料不可用和证据不足状态。 |

### 3.4 工作台页面共同结构

所有 R1 页面低保真稿应包含：

- 统一 App Shell，保持左侧全局导航、右侧页面详情区和右上角用户登录信息区稳定。
- 当前页面标题、业务状态和关键能力标签。
- 主要内容区，优先展示当前页面最需要完成的任务，而不是大面积说明文案。
- 状态区，统一表达加载中、暂无记录、部分资料不可用、加载失败、可重新尝试、部分资料不可见。
- 安全信息边界，明确不展示敏感字段。
- 可进入下一步的主操作，避免只有静态空态。

### 3.5 状态表达

低保真稿必须显式预留以下状态：

| 内部状态 | 页面表达 |
| --- | --- |
| loading | 加载中，使用骨架或轻量提示，不使用空白屏。 |
| empty | 暂无记录 / 暂无复盘，并提供下一步动作。 |
| degraded | 部分资料不可用，结果可信度降低。 |
| failed | 加载失败 / 生成失败，并提供恢复动作。 |
| retryable | 可重新尝试。 |
| permission_filtered / permission-hidden | 部分资料不可见，不泄露对象是否真实存在。 |
| rag_unavailable | 参考资料暂不可用。 |
| index_pending | 资料处理中。 |
| index_failed | 资料处理失败。 |
| no_result | 暂无匹配资料。 |

页面上不得直接出现 `degraded`、`failed`、`empty`、`retryable`、`permission_filtered`、`rag_unavailable`、`index_pending`、`index_failed` 或 `no_result` 等英文技术状态。

### 3.6 安全展示边界

所有页面禁止展示：

- full prompt。
- raw LLM response。
- secret、token、password、provider key。
- 完整数据库连接串。
- object storage path。
- hidden resource id。
- embedding vector。
- provider 原始载荷。

页面可以展示安全摘要、业务状态、综合评分、评分理由、参考来源、证据不足提示、导出版本摘要和用户可理解的失败原因。

## 4. 页面清单

| 页面 | Route | Penpot frame | R1 目标 |
| --- | --- | --- | --- |
| 工作台首页 | `/` | `R1 Workbench Home` | 成为工作台默认首页，汇总主链路入口、最近记录和复盘能力。 |
| 简历和岗位管理 | `/jobs`、`/resumes` | `R1 Jobs and Resumes` | 在统一父栏目下展示岗位和简历列表、创建入口、状态和空态。 |

| 评分 / 复盘详情页 | `/interviews/:sessionId` | `R1 Trusted Review Detail` | 展示可信评分、复盘、参考来源、证据不足、面试过程记录和导出状态。 |

| 状态集合 | 多页面适用 | `R1 Degraded States` | 统一参考资料不可用、问题生成不可用、导出失败、部分资料不可见等状态。 |

旧 W10 mock 不允许作为主界面；只能保留在 `/legacy-mock` 或 `/mock`。

## 5. 页面规格

### 5.1 工作台首页

页面目标：根路径 `/` 的工作台默认首页，帮助用户从一个清晰的工作台进入发起模拟、历史记录、岗位、简历、复盘和可信详情。

必须包含：

| 区块 | 低保真要求 |
| --- | --- |
| App Shell | 左侧全局导航选中“工作台总览”，右侧详情区显示首页内容，右上角显示用户头像、用户名称、个人配置、偏好设置和退出登录入口。 |
| 顶部区域 | 在右侧详情区顶部显示产品名、当前用户摘要、面试准备状态和工作台概览。 |
| 主导航 | 左侧导航展示工作台总览、简历和岗位管理、模拟面试、复盘、知识库；旧 mock 不进入主导航。 |
| 工作台概览 | 展示“数据已保存，可随时回看”“参考资料已关联”“面试过程记录”“综合评分”“复盘结果可查看”，并预留“部分资料不可用”“加载失败”状态位。 |
| 主操作入口 | 首要操作是“发起模拟面试”；次要操作是历史记录、岗位管理、简历管理、复盘。所有入口必须是真实 route，不允许 `#anchor` 假入口。 |
| 最近模拟记录 | 展示记录名称或编号、更新时间、当前状态、过程信息完整度、综合评分、复盘状态、导出状态、证据不足摘要和“查看复盘结果”。 |
| 复盘能力摘要 | 汇总面试过程记录、参考来源、证据不足、综合评分、评分理由、改进建议和 Markdown 导出。 |
| 岗位 / 简历 / 知识库入口 | 首页应显示三个准备对象的可用性摘要，并提供进入对应管理页或发起页的动作。 |
| 历史 / 复盘入口 | 首页应可进入 `/interviews` 和 `/reviews`，并能从最近记录直接进 `/interviews/:sessionId`。 |
| 状态表达 | 首页必须展示最近记录加载失败、暂无记录、早期记录暂无完整过程信息、参考资料暂不可用、暂无匹配资料、导出失败时可重新尝试等低保真状态。 |

Codex 实现注意事项：

- 后续实现不得仅照抄当前 Ant Design card 堆叠；布局以 Penpot 首页 frame 为准。
- 首页可以保留 Ant Design 作为组件基础，但信息架构和视觉密度以 Penpot frame 为准。
- 首页只能展示安全摘要，不展示 prompt、raw response、secret 或私有存储路径。
- 根路径 `/` 必须呈现统一 App Shell，而不是只呈现单页顶部导航。
- 根路径不得出现旧 mock 主界面文案；旧 mock 只能保留在 `/legacy-mock` 或 `/mock`；Penpot 主工作台 frame 不得包含旧 mock 原型文案。

### 5.2 历史列表页

页面目标：`/interviews` 成为模拟记录列表页，支持用户扫描历史记录并进入可信详情。

必须包含：

| 区块 | 低保真要求 |
| --- | --- |
| App Shell | 左侧导航选中“模拟面试 / 模拟面试历史”，右侧详情区显示历史列表，右上角用户区保持可见。 |
| 列表字段 | 记录名称或编号、状态、面试类型、轮次、更新时间、综合评分、复盘状态、导出状态、过程信息完整度、证据不足提示。 |
| 状态标签 | “复盘已生成”“已归档”“可查看”“暂无记录”“部分资料不可用”“加载失败”“可重新尝试”等状态有固定标签位。 |
| 评分摘要 | 展示“综合评分：82”或“暂无评分”；不得把无评分误显示为 0 分。 |
| 复盘状态 | 展示“复盘已生成”“结果可信度降低”“生成失败”“暂无复盘”。 |
| 导出状态 | 展示“导出已生成”“导出失败”“暂无导出”、失败原因和是否可重新尝试。 |
| 点击进入详情 | 每条记录必须有“查看复盘结果”入口，进入 `/interviews/:sessionId` 并保留当前用户上下文。 |
| 空态 | 没有历史记录时，提示用户从发起模拟面试开始，并提供发起入口。 |
| 加载失败态 | API 失败时展示错误摘要和重试 / 返回工作台动作。 |
| 权限不可见态 | 当当前身份无权查看历史时，展示“当前身份暂无可见记录”，不泄露隐藏 session id。 |

Codex 实现注意事项：

- 历史列表不得展示 `unsafe_debug`、prompt、raw LLM response、object storage path 或 hidden resource id。
- 后续若引入筛选 / 分页，不能改变 `/interviews` 作为历史列表的主定位。

### 5.3 评分 / 复盘详情页

页面目标：`/interviews/:sessionId` 展示一场模拟面试的可信评分、复盘、证据链和导出状态。

必须包含：

| 区块 | 低保真要求 |
| --- | --- |
| App Shell | 左侧导航选中“复盘 / 评分 / 复盘详情”或从“模拟面试历史”进入详情时保留合理高亮，右上角用户区保持可见。 |
| 总分 | 显示综合评分、评分状态、低置信度提示；无分数时显示早期记录空态。 |
| 多维评分 | 每个维度显示维度名称、分数、评分理由、参考来源、证据不足、低置信度提示。 |
| 每个维度评分理由 | 评分理由必须和分数并列展示，不允许隐藏到 tooltip 中作为唯一入口。 |
| 参考来源 | 展示资料摘要、片段摘要、位置和引用标记。 |
| 证据不足 | 展示“暂无匹配资料”“部分资料不可见”“资料处理中”“资料处理失败”“参考资料暂不可用”等提示。 |
| 面试过程记录 | 展示面试场次、轮次、回答、评分、复盘、导出记录和过程信息数量。 |
| 复盘摘要 | 展示整体复盘摘要，不与维度评分理由混合。 |
| 改进建议 | 展示可执行建议列表。 |
| 薄弱项 | 展示薄弱项列表，并预留后续训练入口。 |
| 导出状态 | 展示导出状态、失败原因、是否可重新尝试和导出版本摘要。 |
| 部分资料不可用 / 生成失败 / 可重新尝试 | 使用单独状态带汇总，避免散落在页面角落。 |
| 早期记录缺少过程或评分 | 早期记录应显示“早期记录暂无完整过程信息”“暂无评分维度”“暂无复盘结果”等稳定空态。 |
| 敏感字段禁止展示 | 禁止展示 full prompt、raw LLM response、secret、object storage path、hidden resource id。 |

Codex 实现注意事项：

- 详情页应该采用“评分复盘摘要 -> 面试过程记录 / 参考资料 -> 导出记录 / 请求引用摘要”的阅读顺序，具体列数和区域位置以 Penpot frame 为准。
- request refs 只能展示安全 label，不展示 provider 原始请求体。
- 证据不足和结果可信度降低不是错误噪声，应作为可信度信息长期可见。

### 5.4 发起模拟面试页

页面目标：`/interviews/new` 从入口页升级为可被 Penpot 设计约束的启动页。

必须包含：

| 区块 | 低保真要求 |
| --- | --- |
| App Shell | 左侧导航选中“模拟面试 / 发起模拟面试”，右侧详情区显示发起流程，右上角用户区保持可见。 |
| 岗位选择 | 显示最近岗位、岗位状态、必填标记和去新建岗位入口。 |

## 6. Penpot frame 规划

所有 Penpot frame 必须带统一 App Shell：左侧全局导航、右侧页面详情区、右上角用户登录信息区。每个 frame 都必须标注左侧导航选中态、右上角用户区、页面标题、主内容区域、状态区、主要操作、空态 / 失败态 / 降级态。

Penpot frame 文字规则：

1. frame 画布正文只使用用户可见业务语言。
2. 技术术语只能写在 frame 注释、本规格的实现说明或验收规则中。
3. 左侧导航必须使用业务栏目名：工作台总览、简历和岗位管理、模拟面试、复盘、知识库。
4. 右上角用户区使用：用户头像、用户名称、个人配置、偏好设置、退出登录。
5. 状态区使用业务文案，不出现英文技术状态。

| 主要操作 | 重新尝试、继续并提示结果可信度降低、返回工作台、补充资料。 |
| 空态 / 失败态 / 降级态 | 汇总展示失败、可重试、不可用和权限不可见。 |
| 关键文案 | `部分资料不可用`、`证据不足`、`可重新尝试`、`问题生成不可用`、`读取失败`、`当前身份暂无可见记录`。 |
| Codex 实现注意事项 | 降级态不能被隐藏；参考资料检索失败时面试可继续但复盘必须标注证据不足；问题生成服务不可用时不能进入真实生成问题链路。 |

## 7. Penpot MCP 后续工作流

后续使用 Penpot MCP 时必须按以下流程执行：

1. 用户先在 Penpot 中打开目标设计文件，并通过 Penpot MCP Plugin 连接当前项目。
2. Codex 使用 Penpot MCP 读取当前 focused page / current page，确认 page 名称、已有 frame 和 selection。
3. Codex 根据本文档整理将创建的 frame、统一 App Shell、左侧导航选中态、右上角用户区、尺寸、区块、组件层级、命名规则和状态覆盖计划。
4. 创建任何 Penpot frame 前，Codex 必须先输出计划并等待用户确认。
5. 用户确认后，Codex 通过 Penpot MCP 创建低保真 frame；创建时优先使用 board、text、rectangle、group、flex / grid layout 等基础形状，不做高保真视觉。
6. Codex 创建后应读取 frame structure，并在必要时导出 frame 预览用于检查。
7. 用户在 Penpot 中手工调整 frame，包括布局、区块顺序、信息密度和文案。
8. 用户明确确认 frame 后，Codex 再通过 Penpot MCP 读取设计稿结构、导出预览，并把 Penpot frame 作为最高优先级布局依据。
9. Codex 按用户确认的 Penpot frame 修复前端页面。
10. Playwright 增加 screenshot baseline，覆盖首页、历史、详情、空态、业务异常状态和旧 mock 隔离。
11. E2E 必须继续覆盖根路径、历史列表、详情页、复盘列表、空态、部分资料不可用 / 加载失败 / 可重新尝试、敏感字段不展示和旧 mock 隔离。
12. 未经用户确认，Codex 不允许自行改变 Penpot 信息架构、删除用户手工调整、合并 frame 或新增不在本文档范围内的页面。

Penpot MCP 执行注意事项：

- 读取结构时优先使用 `penpotUtils.getPages()`、`penpot.root`、`penpot.selection` 和 `penpotUtils.shapeStructure()`。
- 创建 frame 时按视觉顺序 append child；如果 board 使用 flex / grid layout，优先修改 layout gap / padding，不手动硬改子元素坐标。
- 创建文字元素时避免重复写 shape 名称；Penpot 图层名和画布文字职责应分开。
- 输出给用户的计划必须列出 frame 名称、左侧导航选中态、右上角用户区、页面区块、主要状态和不会修改代码的保证。
- 后续从 Penpot 到代码时，缺失的样式值不能由 Codex 自行发挥，应先标注缺失并向用户确认。

## 8. 后续页面实现验收原则

后续页面实现必须遵守：

- Penpot frame 是最高优先级布局依据。
- 现有文字描述只作为辅助。
- 不能只凭现有代码推断布局。
- 根路径 `/` 必须显示统一 App Shell，并符合 Penpot 首页。
- 左侧全局导航在主要页面保持一致。
- 右上角用户头像、用户名称、个人配置、偏好设置和退出登录入口在主要页面保持一致。
- `/jobs`、`/resumes` 或合并后的“简历和岗位管理”入口必须来自左侧导航。
- `/interviews`、`/interviews/new` 必须来自“模拟面试”栏目。
- `/interviews` 必须符合 Penpot 历史列表。
- `/interviews/:sessionId` 必须符合 Penpot 评分复盘详情。
- `/reviews` 必须来自“复盘”栏目。
- 知识库栏目必须为参考资料、文档资料和资料处理状态保留入口。
- 旧 mock 只能在 `/legacy-mock` 或 `/mock`。
- 旧 mock 不得作为左侧主导航项。
- 不展示 full prompt、raw LLM response、secret、object storage path、hidden resource id。
- 页面主文案不出现数据库、测试、内部 gate、traceability、RAG citation、degraded、failed、empty 等工程术语。
- 用户能理解每个状态的业务含义，状态文案应使用“暂无记录”“资料处理中”“资料处理失败”“参考资料暂不可用”“部分资料不可见”“可重新导出”等表达。
- 工程术语只保留在实现注释、验收说明或开发文档中，不进入页面主文案。
- 旧 mock 文案只作为禁止项，不作为页面内容。
- 根路径不得出现旧 mock 主界面文案。
- 旧 mock 只能保留在 `/legacy-mock` 或 `/mock`。
- Penpot 主工作台 frame 不得包含旧 mock 原型文案。
- 页面能力必须有 Playwright E2E。
- Playwright E2E 后续应断言首页不出现旧 mock 文案。
- Playwright E2E 后续应断言页面出现业务状态文案，例如“暂无记录”“资料处理中”“可重新导出”。
- Playwright E2E 后续必须覆盖左侧导航可见、用户头像可见、用户下拉可打开、左侧导航进入各主要栏目、详情区内容随路由变化。
- 关键页面必须有 screenshot baseline。
- R1 未被用户确认 closeout 前，不得把本设计规格解释为 R1 acceptance closeout 已完成。
