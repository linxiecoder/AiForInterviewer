# Workbench MVP 范围设计

## 1. 文档定位

本文档只保留 Workbench MVP 的设计范围边界，回答“设计层覆盖哪些系统面、哪些系统面暂不设计”。产品范围、需求边界、用户角色和验收口径以 `docs/requirements/workbench-mvp/scope-and-acceptance.md` 为准。

本文档不承载执行日志、迁移过程、任务流水、状态写回或实施授权。

## 2. 设计覆盖面

| design area | design responsibility |
| --- | --- |
| 信息架构 | 定义工作台主要页面、导航关系、用户旅程和信息组织方式 |
| 对象模型 | 定义岗位、简历、材料、面试记录、问答、评分、复盘和弱点训练材料之间的关系 |
| RAG / 上下文 | 定义材料如何作为上下文被检索、引用和传递给 LLM |
| 多轮会话 | 定义会话状态、追问关系、上下文保留和结果沉淀方式 |
| 后端边界 | 定义 API、保存、恢复、评分、复盘和导出所需的服务边界 |
| 评分与复盘 | 定义评分结构、复盘类型、弱点映射和结果交付方式 |
| 质量门禁 | 定义设计层 DoD、错误态、空状态和测试验收输入 |

## 3. 设计非目标

| non-goal | reason |
| --- | --- |
| 企业级组织和复杂权限体系 | 超出一期工作台设计边界 |
| 实时语音 / 视频交互 | 当前设计以文本闭环为主 |
| 完整商业化计费 | 不属于当前设计目标 |
| 大规模运营后台 | 当前只覆盖最小内容维护和验证需求 |
| 正式 implementation packet | 需要 formal window 与状态 gate 另行通过 |

## 4. 与需求层的关系

- 需求层定义 in-scope、out-of-scope、用户角色和验收口径。
- 设计层基于需求层定义信息结构、对象关系、交互路径和工程边界。
- 若设计层与需求层出现职责重叠，需求口径回到 `docs/requirements/workbench-mvp/**`，设计口径保留在本目录。

## 5. 当前限制

本文档不修改 `DOC_STATE.yaml`，不打开 formal window，不生成 implementation packet，不声明任何任务可直接实现。
