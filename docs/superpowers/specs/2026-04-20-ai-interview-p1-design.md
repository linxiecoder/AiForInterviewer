# AI 模拟面试 P1 文本版闭环设计稿

## 1. 文档目标

本文档用于定义 `AI 模拟面试` 项目的 `P1 文本版闭环 MVP` 设计基线，作为后续实现计划、低保真原型和开发落地的输入。

当前文档只覆盖 `P1`，不覆盖语音、视频、多团队增强和复杂运营能力。

## 2. 项目分期

### P1：文本版闭环 MVP

覆盖以下能力：

- 团队管理员 + 成员角色
- 单团队产品形态，数据模型预留多团队扩展
- 岗位管理
- 简历管理
- 岗位与简历匹配分析
- 文本模拟面试
- 打磨模式与模拟模式
- 真实面试复盘与模拟面试复盘
- 薄弱项聚合、训练、消减
- 资产库与资产类型配置
- 管理台模型配置、评分规则配置、成员管理

### P2：多模态面试

- 语音对话
- 视频对话
- 实时转写
- 音视频记录
- 更复杂的面试状态同步

### P3：高级治理与增强

- 更强的资产质量治理
- 搜索源治理与运营
- 多团队增强
- 更复杂的模型策略与成本治理

## 3. 产品边界

### 3.1 用户与团队模型

- 产品角色分为 `团队管理员` 和 `成员`
- 第一版按 `单团队` 产品交付
- 所有核心业务表预留 `team_id`
- 后续可扩为多团队/多租户，但 P1 不实现多团队管理界面

### 3.2 智能能力边界

- P1 采用 `全量 AI 驱动`
- AI 负责：出题、追问、打磨点评、模拟面试节奏控制、岗位匹配分析、复盘生成、改进建议抽取
- 评分体系采用 `规则化评分框架 + AI 解释`
- 检索得到的材料只作为 `Question Context / 参考材料包`
- 参考材料不能作为原题直接复用，最终问题必须由模型结合上下文重新生成

### 3.3 简历处理边界

- 支持导入 `PDF` 和 `MD`
- 导入 `PDF` 时保留原始文件
- 系统异步执行 `PDF -> MD`
- 用户后续主要基于 `MD` 查看、编辑、预览
- 系统支持基于当前 `MD` 导出 `PDF`

### 3.4 面试模式边界

- `打磨模式`
  - 围绕主题持续训练
  - 回答后立即点评
  - 支持重答
  - 支持全场进展与能力树更新
- `模拟模式`
  - 更接近真实面试流程
  - 面试中不进行重干预
  - 结束后给出整场评估、岗位匹配度、通过概率和薄弱点

## 4. 推荐技术方案

P1 推荐采用 `Next.js + FastAPI + PostgreSQL` 的模块化单体架构。

### 4.1 前端

- `Next.js`
- 面向后台工作台、详情页、编辑器、对话页、复盘页、管理台

### 4.2 后端

- `FastAPI`
- 负责鉴权后的业务 API、面试编排、复盘编排、检索编排、任务调度入口

### 4.3 数据层

- `PostgreSQL`
- 启用 `pgvector`
- 保存结构化数据与检索向量

### 4.4 缓存与异步任务

- `Redis`
- 用于缓存、任务状态、异步作业中间态

### 4.5 对象存储

- 保存原始 PDF、MD 快照、导出 PDF、联网搜索原文快照

### 4.6 基础设施补充

- `前端 UI 组件层`
- `日志与可观测性组件`
- `模型注册与推荐组件`

## 6. 问题生成与参考材料原则

系统可使用的参考输入包括：

- 岗位要求
- 简历内容
- 资产库内容
- 历史复盘内容
- 历史面经
- 实时互联网搜索结果

这些内容的角色是 `参考材料包`，不是完整题库。

系统生成问题时采用两步：

1. `考点规划`
   - 决定当前轮次考什么
   - 为什么考
   - 关注哪些风险点和能力点
2. `问题生成`
   - 由模型基于考点规划与当前上下文生成自然问题

日志记录的重点不是“原题来自哪里”，而是：

- 参考了哪些材料
- 命中了哪些考点
- 为什么选择当前追问方向

## 8. 评分与评估体系

采用 `规则化评分框架 + AI 解释`。

### 8.1 题目级评分

维度包括：

- 正确性
- 完整性
- 相关性
- 深度
- 结构表达
- 岗位贴合度
- 风险暴露度

### 8.2 节点级评分

将多题得分累计到能力树节点：

- 技术模块
- 技术点
- 项目主题
- 表达主题

### 8.3 场次级评分

一场面试输出：

- 知识掌握
- 项目可信度
- 表达结构
- 岗位贴合度
- 压力应对
- 总体风险

### 8.4 结果级判断

在场次级评分基础上计算：

- 岗位匹配度
- 薄弱点
- 建议优先补强项
- 通过概率

`通过概率` 必须来自规则计算，不允许作为脱离上下文的独立黑盒数值。

## 9. 核心业务流程

### 9.1 简历导入与编辑

1. 用户上传 PDF 或 MD
2. 若为 PDF，系统保存原件并异步转 MD
3. 用户查看并编辑 MD
4. 保存时生成版本快照
5. 用户可随时导出 PDF

### 9.2 岗位创建与简历绑定

1. 创建岗位
2. 绑定一份或多份简历
3. 在岗位详情查看岗位-简历匹配分析
4. 系统输出匹配度、缺口、高风险项、建议打磨主题

### 9.3 模拟面试启动

1. 在岗位详情或模拟面试模块发起新面试
2. 选择岗位、简历、模式
3. 系统整理参考材料包
4. 系统生成面试策略与首题

### 9.4 打磨模式

1. 支持从以下方式开启：
   - 基于薄弱项
   - 基于自定义主题
   - 基于岗位/简历自动推荐
2. 薄弱项不是必填输入
3. 用户每轮回答后，系统输出：
   - 本题得分
   - 失分点
   - 失分证据
   - 参考回答
   - 为什么这样回答更好
   - 参考回答与失分点对应关系
   - 相关技术原理
4. 系统同步更新：
   - 能力树
   - 当前进展
   - 薄弱点
   - 建议下一题方向

### 9.5 模拟模式

1. 更贴近真实面试节奏
2. 面试过程中弱化中途打断
3. 结束后输出：
   - 最终掌握情况
   - 岗位匹配度
   - 薄弱点
   - 通过概率
   - 建议打磨主题

### 9.6 复盘

复盘来源包括：

- 真实面试材料
- 模拟面试结果

#### 真实面试复盘

要求逐题严格拆解，每题至少包含：

- 原始问题
- 原始回答
- 问题意图
- 回答问题
- 遗漏点
- 错误点
- 表达问题
- 更优回答框架
- 继续追问风险

#### 模拟面试复盘

重点展示：

- 整场判断
- 多维评分
- 岗位匹配度
- 通过概率
- 逐题点评
- 改进建议

### 9.7 归档到资产库

1. 支持整份归档
2. 支持单题归档
3. 归档时选择资产类型
4. 若类型带 schema，则动态渲染字段表单

## 10. 薄弱项体系

### 10.1 定义

薄弱项是 `可训练、可累计、可消减、可停练` 的中粒度训练主题。

薄弱项不是很多小知识点，而是聚合主题，例如：

- `缓存治理与高并发设计薄弱`
- `项目难点讲不深`
- `回答结构混乱`

### 10.2 来源

正式来源包括：

- 岗位-简历匹配分析
- 模拟面试详情/报告
- 复盘报告

### 10.3 聚合原则

- 按岗位聚合
- 按主题归并
- 保留所有证据
- 用户管理的是聚合后的 `WeaknessItem`

### 10.4 状态

- `active`
- `low_priority`
- `dismissed`
- `resolved`

### 10.5 消减规则

当满足以下条件时，可由 `active` 降为 `low_priority`：

- 最近 `2-3` 次相关打磨没有再出现核心失分
- 关联主题平均得分达到阈值
- 能力树节点从 `薄弱` 提升到 `基本掌握`
- 最近一次模拟面试中该主题不再是高风险项

当满足以下条件时，可由 `low_priority` 进入 `resolved`：

- 最近 `2` 场完整模拟面试未再次暴露
- 相关问题通过率稳定
- 岗位匹配分析中的对应缺口已不显著
- 综合评分持续高于通过线

### 10.6 用户停练规则

- 用户可将薄弱项标记为 `dismissed`
- `dismissed` 不等于删除
- 默认不再推荐进入待打磨
- 如后续再次大量暴露，系统可提示恢复为 `active`

## 11. 训练机制

### 11.1 待打磨清单

`待打磨` 是执行层，不等于薄弱项中心。

一个主题可以：

- 归并到薄弱项
- 加入待打磨
- 两者都做
- 只临时加入待打磨

### 11.2 训练抽屉

系统提供统一的 `训练抽屉`，用于在当前页面内完成训练相关操作，不强制跳转。

入口页面：

- 岗位详情
- 模拟面试详情/报告
- 复盘详情

用户点击 `纳入训练` 后，右侧打开抽屉，支持：

- 归并到薄弱项
- 加入待打磨
- 立即发起打磨
- 暂不处理

训练抽屉中还需展示：

- 来源摘要
- 严重度
- 关联岗位
- 证据摘要
- 推荐归并主题
- 操作影响预览

### 13.2 最新模型推荐

系统需结合现实情况推荐最新模型，但不把模型名硬编码为产品常量。

需要维护：

- `provider`
- `model_id`
- `display_name`
- `release_date`
- `capabilities`
- `cost_profile`
- `status`
- `recommended_for`
- `last_verified_at`

系统按任务场景给推荐：

- 文本面试
- 严格复盘
- 长上下文分析
- 成本优先
- 效果优先

## 18. 设计结论

P1 的核心不是单一聊天页面，而是一个围绕 `岗位`、`简历`、`面试`、`复盘`、`薄弱项`、`资产库` 建立的训练闭环系统。

系统的关键产品能力有三条：

1. `基于岗位和简历的动态面试与复盘`
2. `基于聚合薄弱项的持续训练`
3. `基于资产库与历史沉淀的长期能力增长`

后续实现应严格围绕本设计稿展开，不应在 P1 内提前引入语音、视频、多团队运营等超范围能力。

## 19. 视觉风格规范

### 19.1 视觉定位

本产品的视觉母题定义为：

- `面试训练档案室`
- `专业分析工作台`

它不是泛 AI 聊天工具，不是炫技型科技产品，也不是纯 BI 仪表盘。用户进入系统后，应感受到：

- 内容被严肃对待
- 判断有依据
- 训练有持续推进路径
- 页面优先服务阅读、判断、训练、归档

整体气质固定为以下 5 个关键词：

- `可信`
- `冷静`
- `锐利`
- `编辑感`
- `治理感`

界面语言应表现为：

- 页面像 `dossier / case file`
- 模拟面试页像 `转录台 + 旁注分析`
- 复盘页像 `诊断报告 + 证据链`
- 工作台像 `任务与风险总览`
- 管理台像 `审计后台`

明确排除以下方向：

- 霓虹深色 AI 风
- 大面积 glassmorphism
- 过度圆角与漂浮卡片
- 强游戏化训练氛围
- 把整站做成纯 BI 大屏

### 19.2 配色、字体与材质规则

视觉主色采用 `B 明亮稳蓝` 方案，要求整体更明亮，但不能丢失专业性与判断感。

基础颜色如下：

- 画布：`#F7FAFF`
- 卡片：`#FFFFFF`
- 次级底色：`#EFF4FF`
- 选中底色：`#DBEAFE`
- 主文本：`#0F172A`
- 次文本：`#475467`
- 辅助文本：`#667085`
- 分割线：`#D5DCE5`
- 强边框：`#BFC9D4`

功能色如下：

- 品牌主色：`#2563EB`
- 品牌 hover：`#1D4ED8`
- 品牌深色：`#1849A9`
- 训练进展：`#0F766E`
- 成功通过：`#15803D`
- 提醒待处理：`#B54708`
- 风险错误：`#B42318`
- 信息提示：`#0369A1`

使用约束：

- `蓝色` 只承担导航、主操作、链接、选中态
- `青绿` 只承担训练推进、已改善、已解决
- `橙色` 只承担提醒、待处理、边缘风险
- `红色` 只承担高风险、严重问题、失败结果
- 大面积渐变与高饱和铺色不进入 P1 主体界面

字体体系如下：

- 全站 UI：`IBM Plex Sans`
- 报告/章节标题：`Newsreader`
- 系统信息/评分/trace：`IBM Plex Mono`

字体层级建议：

- 页面标题：`32/38`
- 区块标题：`24/30`
- 卡片标题：`18/28`
- 正文：`14/22`
- 辅助文字：`12/18`

材质规则：

- 控件圆角：`10px`
- 卡片圆角：`12px`
- 面板/抽屉圆角：`14px`
- 浮层圆角：`16px`
- 层级依赖 `边框 + 浅底 + 少量阴影`
- 页面卡片不使用重阴影
- 高亮优先使用浅底与左侧强调线，而不是 glow

### 19.3 页面级布局规则

全局版式采用：

- 顶栏
- 左侧导航
- 主内容区
- 可折叠右侧上下文/操作区

但不是所有页面都常驻三栏。P1 页面固定分为 4 种版式：

1. `总览页`
   - 用于工作台
   - 先给任务与风险，再给历史与趋势
2. `列表 + 筛选页`
   - 用于岗位、简历、模拟面试、复盘、资产库、管理台
   - 强调扫描、筛选、进入详情
3. `分屏编辑/分析页`
   - 用于简历编辑、岗位详情、训练相关页面
   - 强调阅读上下文与操作并存
4. `报告/诊断页`
   - 用于复盘详情、模拟面试结果
   - 强调结论、证据、建议

布局约束：

- 详情页优先 `双栏/三栏`
- 编辑页优先 `左编右预览`
- 报告页优先 `顶部结论 + 逐题拆解`
- 同一屏只允许一个视觉主角
- 抽屉用于轻编辑、训练、归档，不用于承载全部核心详情

### 19.4 组件与交互规则

组件语言统一为：

- `边框分层`
- `浅底标注`
- `弱阴影`
- `强结构`
- `弱装饰`

导航与操作：

- 左侧导航选中态使用 `#DBEAFE` 底 + 深蓝文字
- 顶栏保持轻量，不做营销式 hero
- 主按钮只使用品牌蓝，一个区域只保留一个主按钮
- 图标按钮 hover 只做底色变化，不做缩放跳动

表格与列表：

- 列表页优先 `表格 + 工具栏 + 筛选条`
- 表头固定，支持排序、筛选、分页
- 行 hover 使用浅蓝灰底，不使用明显阴影
- 重要字段左对齐，数值与时间右对齐

表单与编辑器：

- 标签始终可见，置于控件上方
- 输入框聚焦态使用浅蓝 focus ring
- Markdown 编辑器更偏文档工作台，而不是表单页

评分与能力组件：

- KPI 卡强调 `指标名 + 当前值 + 相对判断 + 依据入口`
- 能力树更像结构化知识地图，不像游戏技能树
- 训练抽屉采用固定骨架：来源、严重度、岗位、证据、推荐主题、操作影响

消息流与复盘块：

- 模拟面试页使用 `转录式消息块`
- 不采用 IM 聊天气泡风格
- 复盘块采用固定结构：
  - 问题
  - 回答
  - 意图
  - 遗漏点
  - 错误点
  - 更优回答框架

动效规则：

- 仅保留功能性动效
- 时长控制在 `160-240ms`
- 支持 `prefers-reduced-motion`

### 19.8 文案语气规则

全站文案语气固定为：

- `专业`
- `直接`
- `克制`
- `可执行`
- `证据导向`

总准则：

`少煽动，多判断；少口号，多结论；少拟人，多依据。`

文案应该像：

- 专业顾问给出判断
- 编辑给出修改建议
- 分析系统提示下一步
- 治理后台说明规则

文案不应像：

- 聊天机器人陪伴对话
- 营销 landing page 夸张承诺
- 学习产品打鸡血
- 游戏化成长系统

推荐词汇：

- `岗位匹配度`

- `高风险项`

- `待打磨主题`

- `继续追问风险`
- `纳入训练`

避免词汇：

- `冲刺`
- `逆袭`
- `拿下`
- `陪练`
- `秒改`
- `一键变强`
- `AI 教练说`

### 19.9 页面模块优先级排序

模块优先级定义如下：

- `P0`：首屏必须可见，直接影响当前动作
- `P1`：重要，但可放在首屏下半区或次级区域
- `P2`：补充信息，默认折叠、抽屉或二级页

全站总原则：

- 首屏优先放 `下一步动作 + 当前风险 + 核心结论`
- 历史信息、说明性文字、扩展日志默认后移
- 同一屏最多保留一个主任务

示例：

- 工作台 P0：
  - 今日待处理
  - 高风险薄弱项
  - 待复盘项
  - 快速入口
- 岗位详情 P0：
  - 岗位摘要
  - 发起模拟 / 发起打磨
  - 匹配分析结论
  - 高风险项
  - 建议训练主题
- 复盘详情 P0：
  - 总评
  - 匹配度
  - 通过概率
  - 主要风险
  - 纳入训练 / 归档动作

### 19.10 跨页面状态系统与标签体系

统一颜色语义如下：

- `蓝色`：进行中、当前选中、可操作
- `青绿`：训练推进、已改善、已解决
- `绿色`：完成、通过、成功
- `橙色`：待处理、提醒、低确定性风险
- `红色`：高风险、失败、严重问题
- `灰色`：草稿、未开始、已归档、停用

主状态词统一为：

- `未开始`
- `进行中`
- `已完成`
- `待处理`
- `已暂停`
- `已归档`
- `草稿`
- `失败`

风险状态词统一为：

- `高风险`
- `中风险`
- `低风险`
- `需关注`
- `稳定`
- `已改善`

辅助标签用于表达：

- 模式
- 来源
- 类型
- 版本
- 可见性

标签样式规则：

- 统一使用 `浅底 + 深字 + 细边框`
- 高度建议 `22-24px`
- 字号建议 `12px`
- 一个对象最多保留：
  - `1 个主状态`
  - `1 个风险状态`
  - `1 个辅助标签`

### 19.11 数据可视化与评分呈现规则

总原则：

- `先结论，后解释`
- `先比较，后装饰`
- `先证据，后图形`

核心评分统一结构：

```text
当前值 / 等级判断 / 简短解释 / 查看依据
```

推荐图形语言：

- 横向评分条
- 分段概率条
- 双值对比条
- 轻量趋势线
- 状态时间轴
- 能力矩阵
- 证据列表

不建议使用：

- 仪表盘式 gauge
- 大面积环图
- 3D 图表
- 雷达图作为主图

关键指标规则：

- `通过概率`
  - 使用 `横向概率条 + 风险分段`
  - 需显示主要影响因子
- `岗位匹配度`
  - 使用 `总分 + 子维度横条`
  - 子维度控制在 `4-6` 个
- `多维评分`
  - 每个维度同时展示分数、等级、简短诊断
- `能力树`
  - 表达覆盖情况、掌握度与风险节点
  - 不做游戏技能树视觉

数字展示规则：

- 百分比默认 `整数`
- 分数统一 `0-100`
- 变化值使用 `+8 / -5`
- 同类指标不混用不同量纲样式

### 19.12 响应式与设备适配规则

适配策略固定为：

`桌面端完整表达，平板端高效浏览，移动端优先完成轻任务。`

断点建议：

- `1440+`：宽屏桌面
- `1280-1439`：标准桌面
- `1024-1279`：窄桌面 / 横屏平板
- `768-1023`：平板
- `375-767`：手机

全局规则：

- 优先折叠右侧栏，不先压缩主内容
- 列表页优先隐藏低优先级列，而不是全部卡片化
- 手机端优先保留：
  - 主状态
  - 风险等级
  - 当前任务
  - 主操作
  - 核心结论

页面级策略：

- 工作台：手机端退化为任务清单
- 岗位详情：手机端改为摘要 + 分段导航
- 简历编辑：手机端改为 `编辑 / 预览` 切换
- 模拟面试：手机端保留单栏消息流与底部抽屉
- 复盘详情：手机端优先执行摘要，逐题内容改为折叠结构
- 管理台：明确为桌面优先模块

### 19.13 无障碍、可读性与长期使用舒适度

总体准则：

- `先可读，再美观`
- `先可操作，再动效`
- `颜色只辅助，不单独承担信息`

基础要求：

- 正文对比度至少 `4.5:1`
- 大标题与关键数字至少 `3:1`
- 正文不低于 `14px / 22px`
- 手机正文优先 `15-16px / 24px`
- 表格最小字号不低于 `12px`

交互要求：

- 所有可交互元素必须有明确 focus 态
- 关键流程支持键盘操作
- 弹层、抽屉支持 `Esc`
- 焦点切换与返回行为完整

阅读与舒适度要求：

- 长文内容列宽控制在 `680-760px`
- 重要状态必须文字化
- AI 失败状态必须可恢复
- 高密度页面避免持续闪烁、自动滚动与强动效

### 19.14 品牌视觉扩展与首页首屏方向

如果产品需要对外展示，首页与品牌物料也必须延续同一套可信度，而不是额外创造营销人格。

首页首屏只回答 4 个问题：

1. 这是什么产品
2. 它服务谁
3. 它与普通 AI 对话工具有什么不同
4. 用户下一步该点击什么

推荐首屏结构：

```text
顶栏：Logo | 产品能力 | 使用场景 | 设计原则 | CTA
Hero
  左侧：标题 / 机制说明 / CTA / 信任标签
  右侧：工作台、模拟面试、复盘三类界面拼图
```

首屏文案方向建议：

- `把模拟面试、复盘和训练放进同一个工作台`
- `面向真实岗位准备的 AI 面试训练系统`
- `从岗位与简历出发，生成可复盘、可训练的面试闭环`

品牌扩展图形语言建议：

- 细网格
- 页边批注线
- 浅蓝高亮面
- 证据标签
- 真实界面截图

明确避免：

- 抽象发光球
- 深色科幻风
- 夸张渐变与重玻璃拟态
- 只有情绪没有系统结构的品牌表达

### 19.15 最终视觉风格定稿摘要

最终定稿语句如下：

`一套以明亮稳蓝为主色、兼具编辑感与治理感的专业面试训练工作台。`

最终执行关键词：

- 像档案室一样可靠
- 像编辑器一样清晰
- 像分析系统一样可解释
- 像训练系统一样可持续推进

最终明确禁止：

- 深色霓虹 AI 风
- 大面积 glassmorphism
- 过度圆角和漂浮卡片
- 聊天气泡式主界面
- 游戏化成长体系
- 营销口号式文案
- 黑盒评分展示

最终执行准则：

- 首屏优先动作、风险、结论
- 所有评分同时展示值、判断、变化、依据
- 生命周期状态、风险状态、辅助标签严格分离
- 大屏保留结构，小屏保留判断
- 让用户在高密度信息中持续工作，而不是持续忍耐

### 19.16 现代极简深化方向

在 `19.1-19.15` 的视觉基线之上，进一步确认本产品的整站深化方向为：

`编辑档案为表层气质，诊断实验室为交互骨架，现代极简为执行方式。`

该方向不是追加一层装饰风格，而是对整个产品内页体系进行统一深化。最终目标是让产品同时具备：

- `编辑档案感`
  - 适用于简历编辑、复盘报告、逐题拆解、证据链
- `诊断系统感`
  - 适用于岗位分析、评分、训练推进、状态判断
- `现代极简感`
  - 适用于明亮底板、精确留白、低装饰、稳定层级

整站应避免滑向以下两个方向：

- 纯企业后台
  - 过于普通，缺乏复盘与文稿气质
- 纯内容产品
  - 过于“编辑风”，不足以承载训练、评分与治理

整站最终应呈现为：

`一间被数字化的面试训练档案室。`

### 19.17 整站视觉语言细化

#### 19.17.1 页面底板

全站页面都应建立在统一、冷静、明亮的底板上：

- 外层画布：`#F7FAFF`
- 内容面：`#FFFFFF`
- 次级面：`#EFF4FF`
- 页面不使用整块彩色背景
- 页面之间依赖留白、线、编号和层级切换，而不是大面积色块切割

观感目标：

`像在一张冷静、清洁、精密排版的工作台上，摊开不同层级的案卷和分析页。`

#### 19.17.2 网格与留白

建议使用比普通 SaaS 更严格的网格系统：

- 主内容采用 `12 列栅格`
- 页面左右留白稳定
- 关键页面保留固定 `页边批注轨`
- 区块间距优先使用 `24 / 32 / 40` 的稳定节奏
- 避免卡片无限嵌套导致的碎片化结构

设计目标不是“很多白”，而是“白得有秩序”。

#### 19.17.3 页边批注轨

关键页面建议保留一条细窄的 `页边批注轨`，通常位于左侧，少数页面可位于右侧。其职责包括：

- 章节编号
- 当前模块名
- 风险摘要
- 证据计数
- 状态短标签
- 快速定位锚点

批注轨不承担主内容，但它是整站最稳定的识别特征之一，应长期保留。

#### 19.17.4 标题系统

标题系统应具有明显档案感，但必须克制：

- 页面主标题：稳、短、明确
- 模块标题：像章节名，不像营销标语
- 报告型页面的大标题可使用 `Newsreader`
- 普通页面标题优先使用 `IBM Plex Sans`
- 章节编号可稳定出现：
  - `01`
  - `02`
  - `03`

#### 19.17.5 面板与卡片语言

卡片与面板不应像漂浮胶囊，而应像“放在桌面上的分析页”。

推荐保留 3 类核心面板：

- `摘要面板`
  - 用于状态、分数、结论
- `正文面板`
  - 用于 JD、简历、复盘正文、长解释内容
- `操作面板`
  - 用于训练抽屉、筛选器、归档动作、快捷操作

统一原则：

- 卡片边界清楚
- 阴影极轻
- 标题与内容距离克制
- 同一页面里卡片种类不超过 3 种

#### 19.17.6 线条与分隔

整站更依赖“线”而不是“块”：

- 区块之间优先使用细分隔线
- 模块开头可使用短横线或编号线
- 风险块使用左侧强调线
- 证据块使用浅底 + 细边框
- 避免厚边框与重阴影

#### 19.17.7 色彩分布比例

建议整站色彩分布遵循：

- `75%` 中性色
- `15%` 品牌蓝
- `10%` 状态色

目标是让整站保持高级、透气、稳定，而不是被品牌蓝淹没。

#### 19.17.8 背景细节与动效

背景与动效应极轻：

- 背景可保留：
  - 局部浅网格
  - 低对比章节编号水印
  - 非连续浅蓝高亮面
  - 近乎不可感知的纸面噪点
- 不保留：
  - 渐变云
  - 发光团
  - 大面积玻璃糊层
  - 复杂纹样背景

动效应表现为“分析工具动效”：

- 抽屉像案卷被拉开
- 区块出现像章节加载
- 评分条像判断结果被填充
- tabs 切换像索引卡滑动
- hover 只做轻量明暗变化

### 19.19 关键页面高保真结构深化

以下页面描述为执行级高保真结构说明，用于指导后续原型、视觉设计和前端实现。

#### 19.19.1 工作台

工作台首屏建议采用 `1 + 2` 结构：

- 左侧大块 `今日重点卡`
- 右侧上部 `待处理事项`
- 右侧下部 `高风险薄弱项`

`今日重点卡` 应包含：

- 标题：`今日重点`
- 一句系统判断
- 两个主动作：
  - `继续打磨`
  - `查看最新复盘`
- 三个小指标：
  - 待复盘数
  - 高风险主题数
  - 最近模拟次数

首屏下半区建议接入：

- 最近岗位
- 最近模拟面试
- 训练推进

页面目标：

`一打开就知道今天先处理什么，而不是先看一堆图。`

#### 19.19.2 岗位详情

岗位详情建议采用三栏结构，但中栏权重最大：

- 左栏
  - 岗位摘要
  - JD 正文
  - 岗位要求拆解
- 中栏
  - 简历匹配分析
  - 维度拆解
  - 当前缺口
  - 高风险项
- 右栏
  - 建议优先打磨主题
  - 当前岗位薄弱项
  - 最近相关模拟结果
  - 发起模拟 / 发起打磨

中栏顶部必须先给一句明确判断，再给 `4-5` 个横向维度条，例如：

- 技术深度
- 项目可信度
- 表达结构
- 岗位贴合度

页面目标：

`一边看岗位，一边看缺口，一边决定下一步怎么练。`

#### 19.19.3 模拟面试

模拟面试页必须形成 `当前轮次正在发生` 的主轴。

建议结构：

- 顶部固定状态条
  - 模式
  - 岗位
  - 简历
  - 当前轮次
  - 当前进展
  - 结束操作
- 左侧窄栏
  - 轮次索引
  - 关键节点
  - 当前进展树
- 中间主栏
  - 转录流
  - 当前问题
  - 当前回答
  - 输入区
- 右侧分析栏
  - 当前题目意图
  - 能力命中
  - 参考材料摘要
  - 轻量评分
  - 下一题方向

要求：

- 不使用聊天气泡
- 打磨模式允许即时点评片
- 模拟模式弱化中途打断
- 输入区始终是中栏视觉终点

页面目标：

`中间像真实面试记录，右边像专业旁注。`

#### 19.19.4 复盘详情

复盘页应优先呈现 `执行摘要`，再进入逐题内容。

建议结构：

- 顶部执行摘要区
  - 总评
  - 岗位匹配度
  - 通过概率
  - 主要风险
  - 建议优先补强项
  - 纳入训练 / 导出报告
- 左侧主栏
  - 逐题复盘块
- 右侧栏
  - 章节导航
  - 风险汇总
  - 快捷动作

逐题复盘块固定包含：

- 原始问题
- 原始回答
- 问题意图
- 遗漏点
- 错误点
- 更优回答框架
- 继续追问风险

页面目标：

`像一份严肃的复盘诊断报告，而不是聊天记录整理版。`

#### 19.19.5 简历编辑

简历编辑页应成为整站最安静、最适合长时间停留的页面。

建议结构：

- 顶部轻工具条
  - 当前版本
  - 最近保存时间
  - 保存
  - 导出 PDF
  - 查看原始 PDF
- 主体双栏
  - 左侧：Markdown 编辑区
  - 右侧：实时预览区
- 附加折叠区
  - 版本历史
  - 转换日志
  - 导出记录

要求：

- 编辑区边界清楚但不抢戏
- 预览区更像成稿文档
- 版本历史默认不打断主编辑流

页面目标：

`左边在工作，右边像成稿。`

#### 19.19.6 训练中心 / 薄弱项中心

训练中心应像 `训练编排台`，而不是学习打卡页。

建议结构：

- 左栏：主题目录
  - 主题名
  - 当前状态
  - 严重度
  - 最近暴露时间
  - 证据数
- 中栏：主题详情
  - 主题定义
  - 影响说明
  - 最近证据
  - 训练记录
  - 状态变化
- 右栏：动作与流转
  - 加入待打磨
  - 立即发起打磨
  - 标记低优先级
  - 暂停训练
  - 恢复活跃

页面目标：

`这是一个在管理训练主题的工作台，而不是在刷学习进度。`

#### 19.19.7 管理台

管理台应是整站最克制、最冷静的一组页面。

建议通用结构：

- 顶部模块切换
  - 成员管理
  - 资产类型
  - 评分规则
  - 模型配置
  - 系统配置
- 主体采用：
  - 列表
  - 配置表单
  - 版本记录
  - 日志入口

不同模块可采用：

- 模型配置
  - 左：配置列表
  - 中：当前配置详情
  - 右：版本信息 / 生效范围 / 最近变更
- 评分规则
  - 左：规则目录
  - 中：规则项编辑
  - 右：说明、影响范围、测试状态
- 成员管理
  - 顶部搜索与筛选
  - 中部表格
  - 右侧详情抽屉

页面目标：

`看起来真的可治理、可维护、可审计。`

### 19.20 核心组件定稿

整套产品建议固定以下 `6` 类核心组件。

#### 19.20.1 结论摘要卡

它不是 KPI 卡，而是 `判断卡`。

固定结构：

- 左上：标题
- 右上：主状态 / 风险状态
- 中部：核心判断一句话
- 下部：1-2 个关键指标
- 底部：主动作

#### 19.20.2 分析块

它是整站最常见的信息单元。

固定结构：

- 标题
- 判断结论
- 依据拆解
- 风险或影响
- 对应动作

#### 19.20.3 证据块

它用于告诉用户“这个判断不是凭感觉来的”。

固定结构：

- 来源标签
- 证据摘要
- 命中点 / 暴露点
- 查看原文 / 查看上下文

#### 19.20.4 训练抽屉

它是最重要的高频动作容器。

固定结构：

- 顶部：来源、严重度、关联岗位
- 中部：证据摘要、推荐归并主题、影响预览
- 底部：
  - 加入待打磨
  - 纳入薄弱项
  - 立即打磨
  - 暂不处理

#### 19.20.5 转录消息块

它是模拟面试页的核心识别组件。

固定结构：

- 左侧：角色标识 / 轮次索引
- 顶部：题目主题 / 时间 / 当前轮次
- 主体：问题或回答内容
- 底部：轻量标签 / 展开操作

明确要求：

- 不做聊天气泡
- 更像一条条转录记录

#### 19.20.6 复盘诊断块

它是报告页的核心识别组件。

固定结构：

- 顶部：题号 / 问题标题 / 风险标签
- 正文第 1 区：原始问题
- 正文第 2 区：原始回答
- 正文第 3 区：问题意图
- 正文第 4 区：遗漏点 / 错误点
- 正文第 5 区：更优回答框架
- 底部：纳入训练 / 单题归档 / 查看依据

### 19.21 执行级 UI Brief

以下内容可直接作为设计师与前端的统一执行基线。

#### 19.21.1 项目定位

`AI 模拟面试 P1` 是一套面向真实岗位准备的专业面试训练系统，不是聊天型 AI 工具，不是普通招聘 SaaS，也不是学习打卡产品。

#### 19.21.2 一句话目标

把 `岗位分析`、`简历编辑`、`模拟面试`、`逐题复盘`、`薄弱项训练` 和 `归档治理` 放进同一个明亮、克制、可解释的工作台。

#### 19.21.3 用户第一感觉

用户打开后，应首先感受到：

- 这里的判断是有依据的
- 这里的内容是被认真处理的
- 这里的训练是可持续推进的
- 这里不是在“和 AI 聊天”

#### 19.21.4 七个关键页面的主印象

- 工作台：今天先处理什么
- 岗位详情：这个岗位的问题在哪
- 简历编辑：这是一份正在被精修的文稿
- 模拟面试：面试正在发生，并被实时理解
- 复盘详情：这是专业诊断，不是随手点评
- 训练中心：问题正在被系统性消减
- 管理台：系统是可治理、可审计的

#### 19.21.5 明确禁区

- 深色霓虹 AI 风
- 大面积 glassmorphism
- 聊天气泡式主界面
- 游戏化成长体系
- 过度圆角卡片墙
- 只给分数不给依据
- 营销海报式文案

#### 19.21.6 最终设计句

`这不是一个会聊天的 AI 产品，而是一套会判断、会记录、会推进训练的专业面试系统。`
