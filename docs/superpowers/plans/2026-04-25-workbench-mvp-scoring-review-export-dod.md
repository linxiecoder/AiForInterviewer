## 3. 评分体系草案

### 3.1 `ScoreReport`

`ScoreReport` 表示一次打磨阶段总结、压力面整场报告、模拟面试复盘或真实面试复盘中的评分报告。它不是单个 LLM 文案块，而是可回看、可解释、可版本化、可导出的结构化对象。

建议字段族：

| 字段族 | 说明 |
| --- | --- |
| `score_report_id` | 评分报告唯一标识。 |
| `source_type` | `polish_session`、`pressure_session`、`mock_review`、`real_review`。 |
| `source_ref` | 指向 `InterviewSession`、`PolishModeSession`、`PressureInterviewSession`、`ReviewReport`。 |
| `score_version` | 评分规则版本，包含维度集合、权重、解释模板和 LLM / prompt 版本引用。 |
| `total_score` | `0-100` 总分。 |
| `score_level` | 分数等级，例如 `excellent`、`solid`、`borderline`、`risk`。具体阈值待确认。 |
| `dimensions` | `ScoreDimension[]`。 |
| `question_scores` | 题级评分或题级解释证据，是否进入总分待确认。 |
| `evidence_refs` | 绑定岗位、简历、回答、RAG、真实面试材料、复盘项等证据。 |
| `llm_assessment` | LLM 解释、摘要和建议。 |
| `rule_assessment` | 规则侧校验、阈值、缺失输入、低置信度原因。 |
| `confidence` | 评分置信度：`high`、`medium`、`low`。 |
| `revision` | 重算 / 修订记录。 |
| `dispute_state` | 用户或管理员质疑、标记低置信度、请求重算的状态。 |
| `created_at / updated_at` | 生成和修订时间。 |

### 3.2 `ScoreDimension`

`ScoreDimension` 是多维评分的一级维度对象。每个维度必须同时具备分值、解释、证据和失分原因，避免黑盒分数。

建议字段族：

| 字段族 | 说明 |
| --- | --- |
| `dimension_key` | 稳定 key，例如 `technical_depth`、`job_match`。 |
| `label` | 用户可读名称。 |
| `score` | `0-100` 维度分。 |
| `weight` | 维度权重。未确认前只能写入草案。 |
| `rationale` | 为什么得到该分。 |
| `strengths` | 该维度做得好的点。 |
| `loss_points` | 该维度主要失分点。 |
| `evidence_refs` | 支撑该维度判断的证据引用。 |
| `confidence` | 维度置信度。 |
| `rule_flags` | 规则命中的风险、缺失输入或强制扣分点。 |

### 3.3 总分计算方式草案

总分建议采用“维度分加权 + 规则校准 + 低置信度提示”的三段式草案：

1. 基础分：按 `ScoreDimension.score * weight` 汇总到 `0-100`。
2. 规则校准：对缺失岗位、缺失简历、RAG 证据不可用、回答为空、明显跑题、证据不足等情况加 `rule_flags`，但不直接由 LLM 自行改写公式。
3. 低置信度提示：证据不足时不隐藏分数，但必须显示“该评分置信度较低，原因是 ...”，并允许后续重算 / 修订。

维度和权重仍待用户确认。推荐草案是：固定一级维度集合，技术深度 / 岗位匹配略高权重，具体岗位类型权重作为后续扩展。

### 3.4 维度分计算方式草案

维度分不直接等于 LLM 主观打分。建议由两类输入组成：

- 规则输入：是否回答问题、是否覆盖岗位关键要求、是否存在事实错误、是否有证据支撑、是否触发风险项。
- LLM 输入：解释回答质量、抽取亮点 / 缺口、判断表达结构和技术深度、生成改进建议。

规则侧负责边界和一致性，LLM 侧负责语义理解和解释。最终维度分需要保存“规则命中 + LLM 解释 + 证据引用”，而不是只保存分数。

### 3.5 每题得分与总分关系

打磨模式和压力面模式不应完全相同：

- 打磨模式：每题得分优先服务训练进展、失分点定位、能力树更新和下一题建议。是否生成正式 `ScoreReport` 待确认。
- 压力面模式：题级表现应进入整场 `ScoreReport` 的证据层，是否按题级分直接加权进入总分仍待确认。
- 复盘：题级分必须能解释逐题点评，但整场总分应避免被少量题目机械平均误导。

推荐草案是：打磨题级分不直接进入正式总分，压力面题级分进入整场总分，真实面试复盘以逐题拆解和整体报告共同生成评分。

### 3.6 打磨模式得分

打磨模式的本题得分建议定位为“训练反馈分”，用于告诉用户本题回答相对目标主题的完成度。它必须展示失分点、失分证据、参考回答和技术原理，但不默认等同于整场面试通过概率。

用户结束打磨时，可生成阶段性总结：本次打磨覆盖主题、能力树变化、薄弱项变化、进展说明、建议下一步。是否生成正式 `ScoreReport` 待确认。

### 3.7 压力面模式得分

压力面模式的评分建议定位为“整场模拟判断”。题组完成后统一生成：

- `ScoreReport`；
- `FinalAssessment`；
- `JobMatchAssessment`；
- `PassProbability`；
- `WeaknessSummary`；
- `SuggestedPolishTopic[]`；
- `MockInterviewReview`。

压力面过程中默认不做完整即时点评，以保留真实面试节奏。是否每题后展示简短提示仍待用户确认。

### 3.8 RAG evidence 如何进入评分证据

RAG evidence 建议进入评分证据链，但不直接决定分数。它应说明：

- 哪些岗位 / 简历 / 知识库证据被用于出题或评分；
- 用户回答是否准确引用或覆盖了相关证据；
- 评分维度中哪些失分点由证据不足、证据矛盾或关键证据遗漏造成；
- RAG 无命中、检索失败或证据不可见时，评分置信度是否降低。

RAG evidence 是否作为岗位匹配 / 证据充分度维度的重要依据仍待确认。

### 3.9 LLM 评分与规则评分边界

| 边界 | 规则评分负责 | LLM 评分负责 |
| --- | --- | --- |
| 输入完整性 | 岗位、简历、回答、题组、RAG 状态是否存在 | 解释缺失输入对答案质量的影响 |
| 公式一致性 | 维度权重、总分汇总、阈值、版本 | 不能自行改写公式 |
| 证据约束 | 证据 ref 是否存在、是否可见、是否命中 | 判断证据与回答之间的语义关系 |
| 失分分类 | 固定分类、严重度范围、可复用标签 | 抽取具体失分点和解释 |
| 低置信度 | 触发条件、显示状态、是否可重算 | 生成低置信度说明文案 |
| 争议处理 | 记录争议、触发重算、保存版本 | 提供修订建议，不直接覆盖历史版本 |

### 3.10 评分解释文案

评分解释文案建议固定为五段：

1. `总判断`：一句话说明当前表现。
2. `为什么是这个分`：解释总分和主要维度。
3. `主要失分点`：列出按影响排序的失分点。
4. `证据依据`：列出岗位、简历、回答、RAG 或真实面试材料证据。
5. `下一步`：给出打磨主题、薄弱项和训练建议。

文案必须避免“你很差”“完全不行”等无诊断价值表达，应聚焦可改进点和证据。

### 3.11 评分版本

评分版本建议包含：

- `score_schema_version`：维度集合和字段结构版本；
- `score_formula_version`：总分公式、权重和阈值版本；
- `prompt_template_version`：LLM 评分 prompt 版本；
- `model_version`：评分所用模型；
- `rag_policy_version`：RAG 证据使用策略版本；
- `created_by`：系统、用户重算、管理员修订。

### 3.12 评分重算 / 修订

评分重算不应覆盖原报告，应生成 `revision`：

- `recompute_reason`：模型更新、规则更新、证据补充、用户手动请求、管理员修订。
- `previous_report_id`：前一版报告。
- `changed_fields`：变化字段。
- `diff_summary`：分数和解释变化摘要。
- `review_required`：是否需要用户确认新版本。

### 3.13 评分争议 / 低置信度提示

当出现以下情况时，必须提示低置信度或争议：

- 岗位 / 简历缺失或版本不明确；
- RAG 检索失败、无命中或证据权限不可见；
- 用户回答过短、空回答或无法解析；
- LLM 调用失败后使用了降级摘要；
- 评分规则版本被更新；
- 用户或管理员标记“评分不准确”。

低置信度不一定阻断查看报告，但应阻断把该报告作为正式训练闭环的唯一依据。

## 4. 打磨模式评分 / 反馈

### 4.1 `PolishQuestionFeedback`

`PolishQuestionFeedback` 表示打磨模式中单题回答后的完整反馈对象。

建议字段族：

| 字段族 | 说明 |
| --- | --- |
| `feedback_id` | 单题反馈 ID。 |
| `polish_session_id` | 所属打磨会话。 |
| `question_id / answer_id` | 对应问题与回答。 |
| `question_score` | 本题训练反馈分，`0-100`。 |
| `loss_points` | `LossPoint[]`。 |
| `loss_evidence` | `LossEvidence[]`。 |
| `reference_answer` | `ReferenceAnswer`。 |
| `improvement_rationale` | `AnswerImprovementRationale`。 |
| `principle_explanations` | `PrincipleExplanation[]`。 |
| `progress_tree_delta` | 本题对 `ProgressTree` 的更新。 |
| `ability_tree_delta` | 本题对能力树的更新。 |
| `weakness_delta` | 对 `WeaknessItem` 的新增、归并或状态建议。 |
| `next_question_recommendation` | `NextQuestionRecommendation`。 |
| `confidence` | 反馈置信度。 |

### 4.2 `LossPoint`

`LossPoint` 用于描述“为什么失分”。字段建议包括：

- `loss_point_id`；
- `category`：表达结构、技术深度、岗位匹配、证据不足、风险控制、事实错误等；
- `severity`：高 / 中 / 低；
- `description`；
- `impact_on_score`；
- `evidence_refs`；
- `suggested_fix`。

### 4.3 `LossEvidence`

`LossEvidence` 表示支撑失分点的证据，不得只写“AI 认为”。字段建议包括：

- 用户回答片段；
- 对应题目；
- 岗位要求；
- 简历事实；
- RAG 引用；
- 真实面试材料；
- 证据摘要；
- 证据强度；
- 可见范围。

### 4.4 `ReferenceAnswer`

`ReferenceAnswer` 不应只是“标准答案”，而应包含：

- 回答框架；
- 示例回答；
- 关键技术点；
- 与失分点的对应关系；
- 使用哪些证据支撑；
- 适合哪些岗位 / 题型；
- 是否由 LLM 生成；
- 版本信息。

### 4.5 `AnswerImprovementRationale`

该对象解释“为什么这样回答更好”。字段建议包括：

- 更好回答解决了哪些失分点；
- 如何更贴合岗位；
- 如何补足证据；
- 如何降低追问风险；
- 如何提升表达结构；
- 与 `ReferenceAnswer` 的段落映射。

### 4.6 `PrincipleExplanation`

该对象解释相关技术原理。字段建议包括：

- `principle_key`；
- 原理说明；
- 与题目关系；
- 与回答失误关系；
- 推荐学习 / 复习方向；
- 是否可归并为 `WeaknessItem`。

### 4.7 `ProgressTree` 更新

每轮打磨反馈必须产生 `ProgressTree` 更新草案：

- 当前节点掌握度变化；
- 新暴露薄弱节点；
- 已覆盖题型；
- 当前训练阶段；
- 建议下一题方向；
- 是否建议用户结束本轮打磨。

### 4.8 `AbilityTree` 更新

能力树更新建议只写入与本轮回答有关的最小节点变化：

- 相关能力节点；
- 掌握度变化；
- 证据来源；
- 置信度；
- 是否进入长期能力画像。

能力树是否做完整树状页面仍待确认。

### 4.9 `WeaknessItem` 更新

打磨模式可产生三类弱项动作：

1. 新增候选弱项；
2. 归并到已有 `WeaknessItem`；
3. 对已有弱项给出消减建议。

消减是否自动执行仍待用户确认。推荐草案是系统给出建议，用户确认后生效。

### 4.10 `NextQuestionRecommendation`

下一题建议应包含：

- 推荐方向；
- 推荐原因；
- 对应薄弱项；
- 预计难度；
- 是否继续当前主题；
- 是否建议切换主题；
- 是否建议结束本轮。

### 4.11 用户结束时的阶段性打磨总结

用户通过 `UserEndDecision` 结束打磨后，建议生成阶段性总结：

- 本次训练覆盖主题；
- 完成题数；
- 主要进展；
- 仍未解决的失分点；
- `ProgressTree` 和 `AbilityTree` 变化；
- `WeaknessItem` 变化；
- 建议下一次打磨主题；
- 是否生成阶段性 `ScoreReport` 待确认。

### 4.12 打磨模式是否生成 `ScoreReport`

推荐草案是：打磨过程中每题生成训练反馈分，用户结束时生成阶段性轻量 `ScoreReport`。该报告用于训练进展，不直接等同于压力面整场评分或通过概率。

### 4.13 打磨模式是否生成 `SessionRecord`

推荐草案是：生成 `SessionRecord`，但记录类型标为 `polish`，列表中与压力面记录区分展示。它应支持回看、继续训练、归档和导出，但不应被误读为真实压力面模拟成绩。

### 4.14 打磨模式是否可进入 Markdown 导出

推荐草案是：可导出阶段性总结、题级反馈、失分点、参考回答、技术原理和下一题建议；是否导出完整原始回答、RAG 引用和弱项状态变化待确认。

## 5. 压力面模式评分 / 反馈

### 5.1 `PressureInterviewSession`

`PressureInterviewSession` 是压力面模式的会话子对象，强调真实面试节奏、题组完成和结束后整场判断。

建议字段族：

| 字段族 | 说明 |
| --- | --- |
| `pressure_session_id` | 压力面模式会话 ID。 |
| `interview_session_id` | 所属 `InterviewSession`。 |
| `question_set_id` | 所属题组。 |
| `strategy` | 题组策略、难度和题型组合。 |
| `current_question_index` | 当前题目索引。 |
| `status` | `not_started`、`in_progress`、`completed`、`interrupted`、`failed`。 |
| `completion` | `InterviewCompletion`。 |
| `final_assessment` | 结束后最终评估。 |

### 5.2 `InterviewQuestionSet`

题组由系统生成，题目数量、难度、题型组合仍待用户确认。题组字段建议包括：

- `question_set_id`；
- 关联岗位、简历、知识库；
- 题目数量；
- 难度组合；
- 题型组合；
- 生成策略；
- RAG 引用状态；
- 是否允许补题；
- 题组版本。

固定 3 题只能作为候选策略之一，不能写成压力面总规则，更不能写成多轮总规则。

### 5.3 `InterviewCompletion`

`InterviewCompletion` 表示压力面结束事件，字段建议包括：

- `completed_by`：用户完成、系统完成、超时、失败后结束；
- `completed_question_count`；
- `skipped_question_count`；
- `interrupted_reason`；
- `final_generation_status`；
- `score_report_id`；
- `review_report_id`。

### 5.4 `FinalAssessment`

压力面结束后生成最终评估，至少覆盖：

- 最终掌握情况；
- 岗位匹配度；
- 薄弱点；
- 通过概率；
- 建议打磨主题；
- 多维评分摘要；
- 主要证据；
- 低置信度提示。

### 5.5 `ScoreReport`

压力面模式必须生成正式 `ScoreReport` 草案。它以整场题组为来源，题级表现进入总分或证据层的方式待确认。

### 5.6 `WeaknessSummary`

`WeaknessSummary` 汇总整场压力面暴露的薄弱点：

- 薄弱主题；
- 严重度；
- 关联题目；
- 关联证据；
- 是否建议生成 / 归并 `WeaknessItem`；
- 建议训练动作。

### 5.7 `PassProbability`

`PassProbability` 不应是独立玄学数字，应绑定：

- 总分；
- 岗位匹配度；
- 高风险失分点；
- 题组覆盖范围；
- 证据充分度；
- 置信度。

具体概率阈值和文案仍待确认。

### 5.8 `SuggestedPolishTopic`

建议打磨主题应能直接进入训练抽屉：

- 主题名称；
- 触发原因；
- 来源题目；
- 来源证据；
- 推荐优先级；
- 可归并弱项；
- 可加入待打磨；
- 可立即发起打磨。

### 5.9 题组策略

压力面题组策略待确认，草案候选包括：

- 固定 3 题；
- 固定 5 题；
- 按岗位和难度配置题数；
- 自定义策略。

题数策略作为实现级默认值候选保留，进入实现计划前由 W13-E/F 压缩为可执行验收清单。

### 5.10 结束条件

已 confirmed：压力面模式由 `InterviewQuestionSet` 驱动，题目完成后通过 `InterviewCompletion` 结束。仍待确认：

- 题目数量；
- 跳题是否允许；
- 超时是否结束；
- LLM 失败是否可重试；
- 是否允许中途暂停 / 继续；
- 是否每题后展示即时评价。

### 5.11 复盘报告

压力面结束后建议生成 `MockInterviewReview`，包含：

- 整场判断；
- 多维评分；
- 岗位匹配度；
- 通过概率；
- 逐题点评；
- 改进建议；
- 薄弱项汇总；
- 建议打磨主题；
- RAG 引用说明。

### 5.12 与历史记录列表的回写

压力面结束后必须回写历史记录列表：

- 状态变为 `completed` 或 `review_ready`；
- 展示题组数量、完成题数、最终分数、复盘状态；
- 支持查看复盘、查看评分、复制 / Markdown 下载；
- 若评分或复盘失败，列表显示失败状态和重试入口。

## 6. 真实面试复盘

### 6.1 `ReviewReport`

`ReviewReport` 是复盘总对象，统一承接真实面试复盘和模拟面试复盘。

建议字段族：

| 字段族 | 说明 |
| --- | --- |
| `review_report_id` | 复盘报告 ID。 |
| `review_source` | `ReviewSource`。 |
| `source_type` | `real_interview` 或 `mock_interview`。 |
| `question_items` | `QuestionReviewItem[]`。 |
| `overall_assessment` | 整体判断。 |
| `score_report_id` | 关联评分。 |
| `weakness_summary` | 薄弱项汇总。 |
| `training_actions` | 可进入训练抽屉的动作。 |
| `archive_actions` | 可归档到资产库的动作。 |
| `export_snapshot_id` | 最近导出快照。 |

### 6.2 `RealInterviewReview`

真实面试复盘来源是真实面试材料。用户已确认真实面试复盘要求逐题严格拆解，但一期实现深度仍待确认。

最小设计覆盖：

- 手动录入或粘贴原始问题；
- 手动录入或粘贴原始回答；
- 逐题生成问题意图、回答问题、遗漏点、错误点、表达问题、更优回答框架、继续追问风险；
- 生成薄弱项候选和训练建议；
- 支持复制 / Markdown 下载；
- 支持整份或单题归档到资产库。

### 6.3 `QuestionReviewItem`

真实面试逐题拆解必须至少包含：

- 原始问题；
- 原始回答；
- 问题意图；
- 回答问题；
- 遗漏点；
- 错误点；
- 表达问题；
- 更优回答框架；
- 继续追问风险；
- 证据引用；
- 建议打磨主题。

### 6.4 `AnswerIssue`

`AnswerIssue` 汇总当前题的回答问题，包括遗漏、错误、表达和风险。

### 6.5 `MissingPoint`

遗漏点说明用户没有覆盖但岗位 / 问题意图要求覆盖的内容。

### 6.6 `ErrorPoint`

错误点说明事实错误、技术错误、逻辑错误或与简历 / 项目经历矛盾的内容。

### 6.7 `ExpressionIssue`

表达问题说明结构混乱、结论不明确、术语使用不准、冗长、跳步或不回应问题。

### 6.8 `BetterAnswerFramework`

更优回答框架不应替代用户经历，而应给出可迁移结构：

- 开场结论；
- 关键背景；
- 行动过程；
- 技术原理；
- 结果量化；
- 风险和反思；
- 可追问点预防。

### 6.9 `FollowUpRisk`

继续追问风险用于提醒用户真实面试中可能被追问的位置：

- 技术细节追问；
- 项目真实性追问；
- 指标和结果追问；
- 责任边界追问；
- 风险和取舍追问。

### 6.10 真实面试复盘如何生成 `WeaknessItem`

真实面试复盘产生的弱项必须保留证据：

- 来源为 `real_interview_review`；
- 指向具体 `QuestionReviewItem`；
- 记录原始问题和回答摘要；
- 归并到现有 `WeaknessItem` 或生成候选；
- 进入训练抽屉等待用户处理。

## 7. 模拟面试复盘

### 7.1 `MockInterviewReview`

模拟面试复盘来源于 `InterviewSession`、`PressureInterviewSession` 或打磨阶段总结。压力面模式默认生成整场模拟复盘；打磨模式是否生成正式复盘待确认。

模拟面试复盘至少覆盖：

- 整场判断；
- 多维评分；
- 岗位匹配度；
- 通过概率；
- 逐题点评；
- 改进建议。

### 7.2 `OverallAssessment`

`OverallAssessment` 建议包含：

- 一句话结论；
- 当前表现级别；
- 主要优势；
- 主要风险；
- 岗位匹配判断；
- 是否建议进入下一轮真实面试准备；
- 置信度。

### 7.3 逐题点评

逐题点评应包含：

- 题目；
- 用户回答摘要；
- 题目意图；
- 本题得分或评价；
- 失分点；
- 证据；
- 更优回答建议；
- 是否建议打磨。

### 7.4 改进建议

改进建议必须可操作：

- 建议打磨主题；
- 推荐下一题方向；
- 可归并薄弱项；
- 可加入待打磨；
- 可立即发起打磨；
- 可归档为资产。

### 7.5 复盘如何进入训练抽屉

复盘详情中的训练抽屉入口应携带：

- 来源摘要；
- 严重度；
- 关联岗位；
- 证据摘要；
- 推荐归并主题；
- 操作影响预览；
- 四类动作：归并到薄弱项、加入待打磨、立即发起打磨、暂不处理。

### 7.6 复盘如何归档到资产库

复盘支持：

- 整份归档；
- 单题归档；
- 选择资产类型；
- 若资产类型带 schema，动态渲染字段表单；
- 保存来源引用、复盘快照和权限。

动态字段范围仍待用户确认。

### 7.7 复盘如何导出 Markdown

复盘导出应从 `ExportSnapshot` 读取可导出内容，避免临时拼接导致历史不可复现。导出范围待确认，推荐包含完整评分、逐题点评、RAG 引用和训练建议，但原始真实面试材料是否包含需用户确认。

## 8. RAG 引用展示与评分证据

### 8.1 面试台展示

面试台中的 RAG evidence 建议展示：

- 引用标题；
- 来源类型；
- 命中摘要；
- 与当前题目的关系；
- 是否被用于出题；
- 无命中 / 检索失败状态；
- 可继续或需重试提示。

面试台不应把 RAG evidence 做成大段材料浏览区，避免干扰答题节奏。

### 8.2 复盘详情展示

复盘详情中的 RAG evidence 建议展示：

- 每题使用了哪些引用；
- 哪些评分维度引用了证据；
- 失分点是否由证据缺失或证据冲突造成；
- RAG 无命中 / 失败时哪些结论未使用知识库证据；
- 引用是否因权限变化不可见。

### 8.3 进入 `ScoreReport`

`ScoreReport` 中建议保存 `evidence_refs`，每条 evidence 至少包含：

- `evidence_id`；
- `evidence_type`：岗位、简历、回答、知识库、真实面试材料、复盘项；
- `source_ref`；
- `display_title`；
- `summary`；
- `used_for`：出题、追问、评分维度、失分点、复盘结论；
- `confidence`；
- `visibility`。

### 8.4 无命中 / 失败提示

RAG 无命中和失败必须可见：

- 无命中：提示“未找到可用知识库证据，本题基于岗位和简历上下文继续”。
- 检索失败：提示“知识库检索失败，本轮未使用知识库证据，可重试或继续”。
- 权限不可见：提示“部分引用因权限变化不可见，评分置信度降低”。

### 8.5 引用与回答、评分维度、失分证据的关系

引用关系建议采用同一证据模型，不拆成多个不可对齐的字段：

- 用户回答引用或遗漏了哪些 evidence；
- `ScoreDimension` 的分数由哪些 evidence 支撑；
- `LossPoint` 绑定哪些 `LossEvidence`；
- 复盘逐题点评绑定哪些 evidence；
- 导出时保留 evidence 摘要和来源。

## 9. Markdown 导出设计

### 9.1 `ExportSnapshot`

`ExportSnapshot` 表示复制 / Markdown 下载时生成的内容快照。建议字段族：

| 字段族 | 说明 |
| --- | --- |
| `export_snapshot_id` | 导出快照 ID。 |
| `source_type` | `score_report`、`review_report`、`polish_session`、`pressure_session`。 |
| `source_ref` | 来源对象。 |
| `content_version` | 导出内容结构版本。 |
| `markdown_content` | 可下载内容。 |
| `copy_content` | 复制用内容，可短于下载内容。 |
| `included_sections` | 包含章节。 |
| `excluded_sections` | 因权限、范围或用户选择排除的章节。 |
| `created_at` | 生成时间。 |
| `created_by` | 用户或系统。 |
| `permission_context` | 导出时权限上下文。 |

### 9.2 `MarkdownExportContent`

Markdown 下载内容建议结构：

1. 标题和导出时间；
2. 来源信息；
3. 总览结论；
4. `ScoreReport`；
5. 逐题复盘；
6. RAG 引用；
7. 薄弱项和训练建议；
8. 归档建议；
9. 低置信度 / 失败提示；
10. 版本信息。

### 9.3 可复制内容范围

复制内容建议更短，适合粘贴：

- 总览结论；
- 总分 / 维度分；
- 主要失分点；
- 改进建议；
- 建议打磨主题。

是否包含完整逐题点评和 RAG 引用待确认。

### 9.4 下载内容范围

Markdown 下载建议比复制更完整，可包含：

- 完整评分；
- 逐题点评；
- 改进建议；
- RAG 引用；
- 薄弱项和训练建议；
- 导出版本。

是否包含原始回答和真实面试材料原文待确认。

### 9.5 权限限制

导出必须遵守权限：

- 未登录不可导出；
- 无记录读取权限不可导出；
- 无导出权限时只能查看，不显示下载；
- RAG 引用因权限不可见时，只保留“引用不可见”提示；
- 真实面试材料原文是否导出必须由用户确认。

## 10. 薄弱项 / 训练机制 DoD

### 10.1 `WeaknessItem` 生成规则

弱项可由三类来源生成：

1. 岗位-简历匹配分析；
2. 模拟面试详情 / 报告；
3. 复盘报告。

生成规则建议：

- 必须绑定来源证据；
- 按岗位聚合；
- 按主题归并；
- 保留所有证据；
- 用户管理聚合后的 `WeaknessItem`；
- 状态只能在 `active`、`low_priority`、`dismissed`、`resolved` 中流转。

### 10.2 `WeaknessEvidence` 展示

展示字段至少包括：

- 来源类型；
- 来源页面；
- 题目 / 回答摘要；
- 证据摘要；
- 严重度；
- 关联岗位；
- 生成时间；
- 是否进入训练。

### 10.3 `WeaknessSummary`

`WeaknessSummary` 用于复盘和压力面报告中汇总薄弱点：

- 薄弱主题；
- 关联题目；
- 证据数量；
- 严重度；
- 当前状态；
- 建议动作。

### 10.4 训练抽屉入口

训练抽屉入口 confirmed 包括：

- 岗位详情；
- 模拟面试详情 / 报告；
- 复盘详情。

### 10.5 训练抽屉动作

训练抽屉必须支持：

- 归并到薄弱项；
- 加入待打磨；
- 立即发起打磨；
- 暂不处理。

### 10.6 消减规则

消减规则仍待用户确认。推荐草案：

- 系统根据连续训练结果、评分变化、复盘表现和用户完成情况给出消减建议；
- 用户确认后状态从 `active` 降为 `low_priority` 或 `resolved`；
- 用户可停练为 `dismissed`；
- 状态变化保留证据和时间。

### 10.7 能力树展示边界

能力树一期建议只展示与打磨模式相关的最小能力节点、掌握度和变化说明。完整能力图谱、复杂趋势和跨岗位对比不进入默认一期 DoD，除非用户另行确认。

### 10.8 训练进展展示边界

训练进展一期建议展示：

- 当前打磨主题；
- 已完成题数；
- 进展树节点；
- 主要失分点变化；
- 建议下一题方向；
- 是否建议结束本轮。

## 11. 错误态 / 空状态

| 状态 | 用户看到什么 | 是否可重试 | 是否可继续 | 是否保存失败状态 | 是否阻断当前流程 |
| --- | --- | --- | --- | --- | --- |

## 12. 一期 MVP DoD

### 12.1 产品 DoD

- 工作台主流程可闭环。
- 登录 / 权限可用。
- 岗位 / 简历可服务端保存。
- 模拟记录列表可用，且是模拟面试模块默认入口。
- 发起模拟面试可用，支持选择岗位、简历、模式。
- 打磨模式可用，按 `ProgressTree` 驱动并由用户决定继续 / 结束。
- 压力面模式可用，按 `InterviewQuestionSet` 驱动并通过 `InterviewCompletion` 结束。
- RAG / 知识库可用，引用和失败态可见。
- 真实 LLM 出题 / 反馈 / 评分可用。
- `0-100` 多维评分可用。
- 真实面试复盘 / 模拟面试复盘达到用户确认范围。
- Markdown 复制 / 下载可用。
