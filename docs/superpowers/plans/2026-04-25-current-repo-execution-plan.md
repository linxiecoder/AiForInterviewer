# AI 模拟面试系统当前仓库执行计划

## 1. 文档定位

- 本文档是当前仓库现实执行计划入口，用于指导下一阶段主项目设计开发与仓库结构决策。
- 本文档不替代 `DOC-PLAN-P1` 的产品蓝图价值，但优先级高于把 `DOC-PLAN-P1` 直接当作当前仓库实施计划的做法。
- 当前主入口仍包括 `README.md`、`AGENTS.md`、`PLAN_LATEST.md`；本文档负责补足“当前仓库执行计划”这一专用入口。

## 2. 当前仓库定位

- 当前仓库主要承载 `AI 模拟面试系统` 的设计文档、治理状态、`doc_governor` / `doc_governance` 工具链，以及对应测试与验证工具。
- 当前仓库真实可核验的主结构仍是根目录治理文档、`docs/`、`tools/doc_governor/`、`tests/` 与 `requirements.txt`。
- 当前仓库不是已经落地完成的“Web 端目录 / API 端目录 / 基础设施目录”一体化业务实现仓库。

## 3. W0-W7 已完成事项

- `W0`：冻结本轮 P0 对齐范围与窗口边界，明确本轮先做治理链路和主入口收口。
- `W1`：新建 `README.md`，重写 `PLAN_LATEST.md`，补记 `EXECUTION_LOG.md`，把主入口切回“当前仓库真相 + 当前推进计划”。
- `W2`：完成 `doc_governor` 治理文档与 CLI 命令面对齐。

- `W4`：统一 `TEST_POLICY` 与测试守卫规则。

- `W6-B`：完成本轮结论收口保存。

## 4. 当前主项目设计开发的真实前置条件

- 需要先冻结首个 MVP 切片的范围、排除项与验收口径，避免从完整产品蓝图横向铺开。
- 当前正式 requirement 层仍只有 `RQ01`；下一轮应先把首切片稳定映射到模块层与关系层，而不是直接下放到业务代码目录。
- 当前首切片直接涉及 `M03`、`M04`、`M06`、`M07`；`M01` 只在后续确需最小壳层骨架时再作为条件性支撑模块介入。
- 当前正式开窗层仍为空；`M03` 只允许继续引用 `MT03_01 / MT03_03` 作为观察蓝本，不得把旧 `ST03_*` 或观察蓝本直接当作实施入口。

## 5. 当前不直接落未来业务目录结构的原因

- 当前仓库现实并不包含这些目录，直接创建会把未来蓝图伪装成当前事实。
- 当前 `doc_governor` 状态层与主入口层尚未完全同步，如果先落代码目录，会继续放大“蓝图 / 现实 / 状态”三层不一致。
- 当前更缺的是落地路径决策与首批开发边界，而不是先把未来多应用仓库目录一次性铺开。

## 6. 下一步建议

### 6.1 W10-A 已冻结的路线

- 当前确认采用“最小功能切片优先”。
- 当前确认首个切片先围绕“岗位 JD 手工输入 + 简历 Markdown 粘贴/编辑 -> 生成首轮模拟面试问题 -> 记录 1 轮问答 -> 输出简版反馈摘要”收敛。
- 当前不允许直接创建未来业务代码目录；是否进入最小业务骨架，需待后续文档轮完成后再复核。

### 6.2 首切片边界

| 维度 | 冻结口径 |
| --- | --- |
| 输入 | 岗位 JD 文本、简历 Markdown、可选面试方向 |
| 处理 | 最小结构提取、岗位-简历对齐、首轮问题生成、单轮问答记录、简版反馈汇总 |
| 输出 | 首轮问题列表、1 轮问答记录、简版反馈摘要、最小改进建议 |
| 排除 | 多轮会话、长期记忆、检索增强、资产归档、真实面试复盘、训练中心、管理台、完整权限体系、完整 CI/CD、未来业务目录全量落地 |
| 验收 | 先完成文档边界与关系层验收；若后续进入最小代码骨架，再补功能级最小验收 |

### 6.3 下一轮模块与关系补齐范围

- requirement：当前仍以 `RQ01` 作为正式 requirement 入口。
- 直接模块：`M03`、`M04`、`M06`、`M07`。
- 条件性支撑模块：`M01`，仅在后续确需最小壳层 / i18n / 共享页面原语骨架时介入。
- 关系补齐时只允许把 `MT03_01 / MT03_03` 作为 `M03` 的观察蓝本引用；`ST04_01`、`ST04_02`、`ST06_01`、`ST06_02`、`ST07_03` 作为后续设计承接对象；正式开窗层继续保持为空。

### 6.4 后续窗口顺序

1. `W10-B`：细化主项目文档中的首切片边界、排除项与验收口径。
2. `W10-C`：补齐 `RQ01 -> module -> subtask/观察蓝本` 关系，并写死“正式开窗层为空”的当前约束。
3. `W10-D`：只有在 `W10-B / W10-C` 均完成且状态评估仍干净后，才重新判断是否需要最小代码骨架；若条件不满足则取消。
4. `W10-E`：若触发 `W10-D`，再执行最小测试与验证；若未触发，则本轮不进入代码验证。
5. `W10-F`：收口、状态复核与后续写回计划。

补充约束：

- `W10-B` 与 `W10-C` 可以在同一冻结边界下并行推进，但 `W10-D` 必须等待两者都完成。
- 如需使用 `doc_governor`，当前只建议用于文档 round 的 `validate-state`、`evaluate-state`、必要时 `plan-round` / `generate-codex-packet`；不进入 implementation packet。
