# P8 Final Runtime Semantics Audit

## Verdict

WARN

当前 patch 可以支持 `partial_with_deferred_gaps` / C4 runtime foundation partial：`AgentGraphRunnerExecutorAdapter`、facade start/resume/status/timeline/cancel adapter routing、descriptor-supported timeline/cancel fail-closed guard、`AgentRuntimeLoopPolicy`、read-only facade replay、typed handoff envelope、expanded trace DTO 和默认关闭 runtime flag 均有代码与 focused tests 证据。本审计不声明 P8 done，不声明 F8/M8 release，不声明 Phase 11 Supervisor / Orchestrator 或 Phase 12 L5 release gate。

WARN 原因：共享 runtime loop policy 已进入 AgentExecutor adapter、所有当前 facade start surfaces、facade-created resume/replay command boundary、Polish question concrete tool calls、Feedback PR8 provider trace gate 和 Feedback direct in-memory runner entry；facade start 已通过 AgentExecutor adapter，facade resume 已通过 adapter formal-ref guard，facade-created status/timeline/cancel 已通过 adapter read/cancel surface 且保留 owner-scope guard。剩余风险转为 current facade start surfaces 与 Question/Feedback direct paths 之外的 future / indirect graph tool-loop expansion 和 remaining product/future graph paths。generic in-memory runtime resume、Polish question concrete HITL resume 和 Feedback service-backed in-memory runner resume 已验证 checkpoint/strict base version/idempotency/allowed action；typed handoff 已具备 AgentExecutor-bound source-result to target-plan foundation，但还不是产品级 orchestration。

## Runtime findings

1. `AgentRuntimeLoopPolicy` is a valid fail-closed contract for required bounds: it rejects non-positive `max_steps` / `timeout_seconds`, negative `max_retries`, empty or incomplete P8 `stop_conditions`, empty `allowed_tools`, empty `allowed_callers`, and invalid `side_effect_policy`.
2. `AgentGraphRunnerExecutorAdapter` creates an `AgentExecutor`-compatible foundation and rejects missing runtime loop policy at adapter start. It blocks runtime results carrying `formal_refs`, blocks result/status/timeline metadata that exposes non-empty `formal_refs` or non-zero formal-write counters including common counter key variants, and blocks replay results that are not read-only/formal-write-blocked.
3. The active production-facing facade now starts all current facade start surfaces by constructing `AgentExecutionPlan` and routing through `AgentGraphRunnerExecutorAdapter`; facade-created status/timeline/cancel for known runs also route through the adapter read/cancel surface with a facade owner-scope guard before adapter access, and facade-created timeline/cancel fail closed before runner access when the known run descriptor lacks the requested entrypoint. The start path injects validated `runtime_loop_policy` metadata with `max_steps`, `max_retries`, `timeout_seconds`, `stop_conditions`, `allowed_tools`, `allowed_callers` and `side_effect_policy`.
4. Polish question graph has pre-existing local loop bounds and tests for timeout, max steps, retries and no false success. Polish question concrete tool calls now require registered runtime `ToolDefinition` lookup and enforce loop-policy caller / side-effect compatibility plus tool-declared forbidden-data payload blocking before execution; the Feedback PR8 provider trace gate also requires registered runtime `ToolDefinition` lookup and now proves caller / permission / owner / side-effect / tool-declared forbidden-data fail-closed behavior through the graph trace request path, and current direct production `authorize_tool_call()` call sites are statically gated. This is useful evidence, but it is not equivalent to full product runtime migration.
5. `AgentSideEffectGuard.authorize_tool_call()` now requires a registered `ToolDefinition` and checks registered tool name, allowed caller, permission scope, owner scope, side-effect policy, global sensitive payloads and recursive `ToolDefinition.forbidden_data` payload keys. Polish question direct runtime policy also includes `interrupt_required` in its stop-condition set and fails closed when a narrowed loop policy excludes the question agent or candidate-write tool. Feedback direct in-memory runner start now fails closed when descriptor loop-policy metadata is absent or mismatched. Future / indirect graph tool-loop expansion outside covered facade start surfaces and Question/Feedback direct paths still needs shared policy / registry consumption evidence.
6. Replay semantics are the strongest runtime area in this patch: facade and adapter both fail closed on non-read-only or formal-write-enabled replay; concrete in-memory, Question and Feedback replay return trace/timeline refs only, preserve original failed/blocked/interrupted/cancelled status category as `replayed_*`, expose refs-only trace comparison metadata including Feedback validation refs, report zero provider/tool/repository/DB/formal-write counters, and the replay DTO rejects positive or unparseable side-effect counters before adapter/facade surfaces can treat them as valid replay.
7. Resume/interrupt semantics remain partial. `AgentInterruptService` validates owner scope, base version, allowed action and idempotency; `AiOrchestrationFacade.resume_interrupted_run()` now carries checkpoint/strict-base/idempotency control fields, validates allowed action before runner invocation and blocks returned runtime formal refs through adapter; service-backed runtime resume now routes generic in-memory user-confirmation, Polish question concrete HITL and Feedback in-memory five-trigger HITL resume through that validation before graph continuation, with Feedback resume timeline preserving candidate / validation refs; generic and Polish question cancel paths record refs-only `run_cancelled` timeline events. Remaining product/future graph paths still do not prove complete schema-bound resume coverage for every interrupt path.
8. Failure status handling is guarded in Question integration tests, runtime DTOs now classify canonical status categories and reject unknown status or success-like status with `failure_reason`, Polish question application task status mapping reuses that taxonomy for existing `AiTaskStatus` projection, adapter result metadata now rejects fake-provider use and fail-open fallback reported as generated success, `AgentTraceBridge` rejects unknown non-DTO trace statuses, and adapter phase/tool metadata event statuses fail closed before trace event emission. Broader DB persistence/API status taxonomy remains deferred.
9. Trace/handoff work is still foundation-level. `AgentExecutionTrace` and `AgentHandoffEnvelope` now carry required fields and recursively filter nested forbidden metadata keys, adapter traces/timeline traces merge command metadata and nested runtime `metadata.trace_refs` `tool_refs` / `policy_refs` / `provider_refs` / `validation_refs` / `handoff_refs` / `low_confidence_refs` with runtime result / timeline event refs, the adapter exposes mapped handoff refs on result/status surfaces plus refs-only handoff candidate descriptors including asset body refs, `build_agent_handoff_plan()` creates target AgentExecutionPlan input refs through the handoff contract, target executor timeline mapping surfaces command handoff envelope refs, Feedback status trace refs include validation refs for replay comparison, and generic runtime start/resume/cancel events plus Feedback service-backed resume/cancel events plus Question/Feedback concrete runtime start events and Question cancel events emit refs-only metadata with generic direct start/resume/succeeded preserving P8 command refs and Question cancel checkpoint/validation refs separated; remaining product/future runtime events still need proof of the full P8 trace matrix.
