# PRD

本文档是 AiForInterviewer MVP 产品需求唯一事实源。历史需求、归档材料和迁移前设计只作为来源证据；进入本文件后才成为 active 产品需求。

## 1. 产品定位

AiForInterviewer 是面向真实岗位准备的 AI 模拟面试工作台。它不是通用聊天工具，也不是招聘管理系统；核心价值是围绕岗位、简历、参考材料、模拟问答、评分复盘和弱项训练，形成可保存、可回看、可改进的文本面试闭环。

## 2. MVP 目标

MVP 必须让用户完成一次端到端文本模拟面试：

1. 录入岗位和候选人资料。
2. 录入或解析简历。
3. 基于岗位、简历和参考材料启动模拟面试。
4. 由 LLM 生成不直接照搬参考材料的问题和追问。
5. 用户提交回答并保存面试过程。
6. 系统生成可解释评分、通过概率、弱项和复盘报告。
7. 用户可回看历史记录，并复制或下载 Markdown 复盘结果。
8. 用户可把弱项进入后续训练闭环。

## 3. 用户角色与权限边界

| 角色 | MVP 权限边界 |
|---|---|
| 练习者 / 成员 | 管理自己的岗位、简历、模拟面试、复盘、导出和训练动作 |
| 管理员 / 内容维护者 | 维护公共参考材料、基础评分口径、账号可见性和最小管理配置 |
| 项目维护者 | 验证范围、验收口径、归档状态和后续任务入口 |

MVP 采用单工作区产品形态，不提供多团队切换、多租户管理或复杂组织后台。数据隔离字段、审计字段和未来团队扩展点进入 F4 技术设计，不在 F1 PRD 中冻结数据库实现。

## 4. MVP 核心用户流程

1. 用户进入工作台，查看最近模拟、待复盘项、训练建议和资料状态。
2. 用户创建岗位，录入公司、岗位名、JD、岗位要求和来源链接。
3. 用户创建或导入简历，完成可编辑文本版本。
4. 用户选择岗位、简历、参考材料和面试模式，启动模拟面试。
5. 系统生成参考材料包、考点规划和首题。
6. 用户完成多轮回答、追问、暂停或结束。
7. 系统保存面试记录、问题、回答、引用证据、低置信度和生成元信息。
8. 系统生成评分、通过概率、弱项和复盘报告。
9. 用户查看复盘，复制或下载 Markdown，并将弱项纳入训练。

## 5. 简历输入与解析范围

| 需求 | 优先级 | 说明 |
|---|---|---|
| 支持 Markdown / 文本简历输入与编辑 | MUST | 用户必须能直接维护可被面试和复盘使用的简历正文 |
| 支持 PDF 简历上传并保留原件 | MUST | 原件作为历史材料和解析来源保存 |
| 支持 PDF 到可编辑文本的解析流程 | SHOULD | 解析失败时必须允许用户手动编辑文本版本 |
| 支持简历版本快照 | SHOULD | 保存历史版本，便于回看和复盘追踪 |
| 支持基于当前文本导出简历 PDF | COULD | 不阻塞 MVP；复盘 Markdown 导出优先级更高 |

## 6. JD / 岗位 / 候选人资料输入规则

岗位输入至少包含公司、岗位名、JD 正文、关键要求、来源链接和状态。候选人资料以简历正文为主，可补充项目经历、目标岗位偏好和自定义上下文材料。

岗位、简历和参考材料必须能组成一次模拟面试的 `LaunchContext`。资料缺失时允许启动，但必须在面试台和复盘中提示证据不足或可信度降低。

## 7. 面试题生成原则

1. 参考材料包只作为上下文，不是题库。
2. 系统不得直接复用参考材料、历史面经或资料原题作为最终问题。
3. 问题生成必须至少包含“考点规划”和“问题生成”两个逻辑步骤。
4. 考点规划需要说明本轮考察什么、为什么考、关注哪些风险点。
5. 问题生成需要结合岗位、简历、历史回答和参考材料生成自然问题。
6. 日志或可追踪记录应保存参考材料摘要、命中考点、追问原因、模型和 prompt 版本。
7. 实时互联网搜索不进入 MVP 必做范围，作为后续能力处理。

## 8. 面试执行流程

MVP 必须支持文本模拟面试主流程。模拟模式接近真实面试节奏，面试过程中弱化中途干预，结束后统一生成整场评估。

打磨模式作为 SHOULD 能力：围绕薄弱项、手动主题或岗位缺口持续训练，允许回答后即时点评、重答和能力进展更新。若首版无法完整实现打磨模式，PRD 仍要求保留弱项到训练入口的可追踪闭环。

面试过程必须支持暂停、继续、结束、失败重试和历史回看。LLM、RAG 或解析失败时，用户应得到明确提示，已保存内容不得丢失。

## 9. 评分、通过概率、弱项识别规则

评分采用规则化评分框架加 AI 解释，不允许只输出黑盒结论。

| 评分层级 | MVP 要求 |
|---|---|
| 题目级评分 | 输出正确性、完整性、相关性、深度、结构表达、岗位贴合度和风险暴露度 |
| 场次级评分 | 输出知识掌握、项目可信度、表达结构、岗位贴合度、压力应对和总体风险 |
| 通过概率 | 必须基于评分、岗位匹配、风险项和证据充分度计算，并展示主要影响因子 |
| 弱项识别 | 从回答缺口、复盘证据、岗位匹配缺口和训练反馈中聚合中粒度训练主题 |

低置信度评分必须显式标注原因，例如材料不足、引用缺失、回答过短或模型判断不充分。

## 10. 复盘报告需求

复盘报告必须覆盖：

- 整场结论。
- 0-100 总分和多维评分。
- 岗位匹配度和通过概率。
- 每个关键问题的原始问题、原始回答、问题意图、遗漏点、错误点、表达问题和更优回答框架。
- 引用证据、证据缺口和低置信度说明。
- 改进建议、弱项和下一轮训练建议。
- Markdown 复制或下载。

真实面试复盘作为 SHOULD 能力，支持用户输入真实面试材料并由 LLM 识别问答边界；低置信度时必须提示人工校对。

## 11. 弱项训练闭环

`WeaknessItem` 是可训练、可累计、可消减、可停练的中粒度训练主题。MVP 必须从复盘结果中生成弱项建议，并允许用户将弱项纳入训练。

MVP 最小状态包括：

- `active`
- `low_priority`
- `dismissed`
- `resolved`

用户必须能查看弱项来源证据、关联岗位、严重度和建议训练方式。自动消减规则、长期训练任务队列和高级训练看板为 SHOULD 或 LATER，不能阻塞 MVP 主链路。

## 12. 管理台能力边界

MVP 管理能力只保留最小边界：

| 能力 | 优先级 | 边界 |
|---|---|---|
| 基础账号和可见性管理 | MUST | 支持最小管理员/成员边界，不提供复杂组织管理 |
| 公共参考材料维护 | SHOULD | 可维护公共知识材料或资料入口，但不做高级质量治理 |
| 评分规则和 prompt 版本说明 | SHOULD | 能追踪当前使用口径；完整可视化配置后台后置 |
| 模型配置 | COULD | 可通过环境配置或后续管理台承载，不作为首版业务阻塞 |
| 成本、搜索策略和运营治理 | LATER | 不进入 MVP 发布阻塞范围 |

## 13. MVP 非目标范围

- 多团队 / 多租户管理。
- 实时语音和视频面试。
- ATS 集成。
- 完整商业化计费。
- 高级检索质量平台。
- 复杂团队共享知识库治理。
- 完整运营后台。
- 批量导出中心和高级资产打包。
- 将模型名、模型版本或第三方 provider 固化为产品常量。

## 14. 需求优先级

| 优先级 | 需求 |
|---|---|
| MUST | 文本模拟面试主闭环；岗位和简历输入；LLM 问题生成；服务端保存；可解释评分；通过概率；弱项识别；复盘报告；Markdown 导出；历史回看；基础权限边界 |
| SHOULD | PDF 解析；打磨模式；真实面试复盘；训练入口；公共参考材料；简历版本快照；评分和 prompt 版本追踪 |
| COULD | 简历 PDF 导出；模型配置管理台；资产库复杂字段；自动弱项消减；更完整训练抽屉 |
| LATER | 多团队、多租户、语音、视频、ATS、商业化计费、高级检索治理、批量导出、高级运营后台 |

1. `docs/01-product/PRD.md` 已创建，并成为 MVP 产品需求唯一事实源。

3. 高风险历史需求没有未分类状态。
4. F2-F8 后续任务入口已进入 `docs/03-delivery/BACKLOG.md`。
5. `docs/03-delivery/DELIVERY_PLAN.md` 已更新 F1 阶段、M1 里程碑和退出标准。
6. active docs 中不恢复旧阶段体系、旧任务体系或 archive 执行依据。
