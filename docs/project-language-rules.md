# 项目语言规范

## 1. 目的

本文档用于统一当前项目的语言使用规范，降低多人协作、文档沉淀、界面表达和后续维护中的歧义。

默认情况下，如果没有用户额外说明，所有内容都应遵循本文档。

## 2. 总原则

当前项目采用以下总原则：

- 文档主体使用中文
- 界面文案使用中文
- 代码本体按常规技术规则处理
- 代码中的注释和说明文字使用中文
- 技术键名、路径、标识符保持英文

## 3. 适用范围

本规范适用于以下内容：

- `README`
- 设计稿
- 实现计划
- 说明文档
- 操作手册
- 接口文档
- Issue
- PR 说明
- 评审说明
- 提交信息
- 界面文案
- 代码注释
- 说明性 docstring

## 4. 文档语言规则

### 4.1 文档主体必须使用中文

所有正式文档的正文、章节标题、说明段落、列表内容默认使用中文。

示例包括：

- 项目说明
- 设计文档
- 开发计划
- 使用说明
- 测试说明

### 4.2 英文只用于必要技术对象

当内容涉及技术对象时，可保留原始英文名称，但整体句子仍应使用中文表达。

例如：

- 模型名
- 框架名
- 三方服务名
- 官方协议名
- 官方 API 名称

推荐写法：

- 使用 `FastAPI` 搭建后端服务
- 使用 `Next.js` 承载前端工作台
- 使用 `GPT-5.4` 作为面试编排模型

## 5. 代码与注释规则

### 5.1 代码本体不强制中文

以下内容按代码规则处理，默认保持英文或技术原语：

- 变量名
- 函数名
- 类名
- 接口名
- 数据库字段名
- 配置键名
- JSON / YAML 键名
- 协议字段

### 5.2 代码注释必须使用中文

代码中的解释性内容默认使用中文，包括：

- 行内注释
- 块注释
- 说明性 docstring
- 示例解释文字

目标是让项目中的说明性语义对中文使用者一致可读。

## 6. 路径与技术标识规则

以下内容保持英文，不强制中文化：

- 目录名
- 文件路径
- URL 路径
- slug
- 包名
- 模块名
- 命令名
- 环境变量名

原因是这些内容直接参与工程结构、命令执行或外部系统兼容，不适合作为中文化对象。

## 7. 界面文案规则

所有面向用户展示的 UI 文案统一使用中文，包括：

- 按钮
- 菜单
- 导航
- 表头
- 表单标签
- 提示语
- 空态文案
- 错误提示
- 成功提示
- 弹窗文案
- 状态标签

如果界面中必须展示英文术语，应以中文界面语义为主，英文术语作为技术名词保留。

## 8. 代码块规则

Markdown 中的代码块整体按代码规则处理，不强制中文化。

适用类型包括：

- `bash`
- `json`
- `yaml`
- `sql`
- `python`
- `typescript`

但如果代码块内部出现注释、解释性说明文字，仍应尽量使用中文。

## 9. 提交与协作文案规则

以下协作内容默认使用中文：

- Git 提交信息
- Issue 描述
- PR 说明
- 评审意见
- 变更说明

如果需要引用英文技术对象，可保留技术名词原文，但整体表达仍以中文为主。

## 10. 执行规则

- 如果后续没有特别说明，默认遵循本规范。
- 如果用户提出新的语言要求，以用户最新要求为准。
- 如果新增正式文档，应同步在根目录 `AGENTS.md` 中建立索引。
- 如果文档与本规范冲突，应优先修正文档使其符合本规范。
