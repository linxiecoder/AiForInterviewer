## 2. 旧切分入口的依赖问题

| 旧入口 | 被混在一起的依赖 | 为什么会导致错误进入子任务阶段 |
| --- | --- | --- |
| `ST03_01` | `jobs` 领域契约 + `OQ-020` + `OQ-021` + M02 页面可见性语义 + 页面实施骨架 | 一个入口同时吃掉领域、列表、详情页和共享 UI / list 契约，无法形成可验证、可并行的子任务边界 |
| `ST03_02` | 简历聚合根 + 版本不变量 + `OQ-006` + `OQ-020` + `OQ-021` + 编辑器页面交互 | 一个入口同时承担领域不变量和页面交互，导致“只看标题像子任务，实际仍是模块层设计包” |
| `ST03_03` | `OQ-007` + M01 对象存储 / 共享下载 + 转换 / 导出状态机 + 业务下载入口投影 | 一个入口跨越上传、转换、导出三条链路，既不能并行，也无法在上游不稳定时形成独立验证点 |

## 5. 面向 M04-M06 的输出 readiness

### 5.1 面向 M04
- 当前只可作为模块设计输入引用：
  - `jobs.id`
  - `jobs.jd_markdown`
  - `jobs.requirement_items_json[*].item_key`
  - `jobs.requirement_items_json[*].text`
  - `jobs.requirement_items_json` 的数组顺序
  - `resumes.id`
  - `resumes.current_document_id`
  - `resume_documents.markdown_content`
- 当前仍不能宣称为完整稳定输入：
  - `jobs.requirement_items_json` 的扩展字段
  - 原因：`OQ-025` 虽已在总控层登记为 `proposed-default`，但当前仍只冻结最小 item 子集、`null / []` 语义、数组顺序与写入责任
- 不应从 M03 引用：
  - `job_resume_bindings`
  - `job_resume_match_analyses`
  - 训练证据对象

### 5.2 面向 M05
- 当前可按现有模块边界引用：
  - 原始 PDF / 导出 PDF 对应的对象引用关系
  - `resume_conversion_logs`
  - `resume_export_records`
- 当前仍需保留风险位：
  - 上传、导出与共享下载网关的实现投影细节
- 需要显式遵守：
  - `resume_documents` 不进入共享 `retrieval_chunks`
  - M05 若要归档简历相关内容，应基于业务显式选择的导出产物或归档流程，而不是直接把当前简历正文当共享知识资产

### 5.3 面向 M06
- 当前只可作为模块设计输入引用：
  - `jobs.jd_markdown`
  - `jobs.requirement_items_json[*].item_key`
  - `jobs.requirement_items_json[*].text`
  - `jobs.requirement_items_json` 的数组顺序
  - `resumes.current_document_id`
  - `resume_documents.markdown_content`
- 当前仍不能宣称为完整稳定输入：
  - `jobs.requirement_items_json` 的扩展字段
  - 原因：与 M04 一致，当前只有 `OQ-025` 的最小共享输入，不包含扩展字段与完整岗位链 readiness
- 需要显式遵守：
  - M06 消费的是“当前简历正文”，不是转换日志内部细节
  - M06 不应依赖 `summary_json` 这类未冻结内部 JSON 结构
- `OQ-025` 的稳定吸收只收窄了岗位侧最小输入，不改变上传 / 导出链对 M01 与 `OQ-007` 的依赖结论。

## 6. 依赖门槛判断

- 进入 M03 模块评审：
  - 当前已满足
  - 原因：全局默认口径与当前模块文档已足以支撑 reviewable 级评审
- 进入 M03 微任务设计：
  - 当前仍不满足“批量开启”
  - 主要原因：
    - `OQ-024` 的固定映射虽已吸收到模块层，但正式开窗层当前为空；这里的“为空”不是缺少候选命名，而是现有 `MT03_*` 中，`MT03_01` / `MT03_03` 仍只满足观察面输入，`MT03_02A` / `MT03_02B` / `MT03_05` 仍受 `OQ-020` / `OQ-021` 的实现级 shared contract 约束，`MT03_06` / `MT03_07` / `MT03_08` 仍受 `OQ-007` 与 M01 共享下载 / 对象存储口径约束；因此还没有任何一项同时满足“正式候选输入 + 允许开窗”
    - 当前阶段仍定义为“白名单观察准备”，不是“正式放行子任务设计”；关窗本身就是直接阻塞，在关窗状态结束前，观察面不得外推为正式候选
    - 当前最低位文档仍是高 `L4` 的 `MODULE_API_DESIGN.md`，但这是开窗层为空、当前阶段关窗、上传 / 导出链依赖未变共同回压后的结果态；单独修改 API 文案不能替代依赖收口
  - 局部判断：
    - `MT03_01`、`MT03_03` 当前只可作为白名单观察面继续复核，不得据此启动正式子任务设计窗口
- 进入 M03 微任务实施：
  - 当前不满足
  - 原因：
    - 还没有新的微任务双文档
    - 上游共享契约尚未全部稳定
    - 当前子任务骨架仍存在模板与成熟度问题

## 7. 当前依赖风险

- M01 仍只给出了方向性基础设施约束，尚未提供足以支撑上传 / 导出微任务的高成熟度下载与对象存储口径。
- M02 已给出权限矩阵基线，但页面级和接口级的最终细节仍不适合在 M03 内部重写。
- `jobs.requirement_items_json` 已按 `OQ-025` 吸收到模块层最小共享输入，但扩展字段仍不可依赖；若总控改写默认口径或调整下游措辞，岗位链 `MT03_01` / `MT03_02A` / `MT03_02B` 需同步回看。
- 不得把 `OQ-025` 的稳定吸收误读成上传 / 导出微任务也已获得足够输入；这两条链仍主要受 `OQ-007` 与 M01 共享下载 / 对象存储口径影响。
- `OQ-024` 已写死“历史容器 / 观察蓝本 / 正式开窗层为空”的三层映射；在当前阶段仍关窗、且最低位 API 仍为高 `L4` 前，不得把 `MT03_01` / `MT03_03` 误判为正式候选。
- “正式开窗层为空”与“当前阶段关窗”是两个并列阻塞：前者说明当前没有可被放行的正式入口，后者说明即使局部输入已稳定也只能停留在观察面；两者缺一不可，且都会回压 `MT03_01` / `MT03_03` 的放行判断。
- 上传 / 导出链依赖之所以仍未变，是因为当前新增事实只发生在岗位侧最小共享输入与列表最小映射吸收，`OQ-007` 与 M01 共享下载 / 对象存储口径本身没有出现新的成熟度跃迁；因此 `upload-pdf`、`export-pdf`、`original-pdf` 仍继续代表整条上传 / 导出链的共享依赖风险。
- `MODULE_API_DESIGN.md` 当前停在高 `L4` 只是一种依赖结果态：只要正式开窗层仍空、关窗未解除、上传 / 导出链依赖未收口，这份最低位文档就会继续被回压在高 `L4`。
- `OQ-021` 与 `OQ-020` 当前都只应被视为 shared contract 候选输入；其中 `updated_after / updated_before` 只属于模块级扩展，不足以让旧 `ST03_01` / `ST03_02` 或新的白名单观察面直接开窗。
- 如果后续 `OQ-006`、`OQ-007` 或 M01 共享下载口径被改写，M03 的上传 / 导出 / 预览相关微任务切分需要整体回看。
