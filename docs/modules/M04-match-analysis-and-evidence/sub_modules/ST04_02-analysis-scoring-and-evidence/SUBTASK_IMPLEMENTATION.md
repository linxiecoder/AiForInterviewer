实施完成后，必须做代码检视，至少检查：

- 是否超出当前子任务边界
- 是否引入无关文件改动
- 是否遗漏异常路径
- 是否违反接口 / schema / logic 契约
- 是否存在明显重复逻辑
- 是否存在命名和可维护性问题
- 是否存在权限 / 安全风险
- 是否存在测试遗漏

## 15. 完成定义（DoD）

当前子任务只有在以下条件都满足时才算完成：

- `SUBTASK_DESIGN.md` 已足够稳定
- 本轮实施目标已全部完成
- 修改范围未超边界
- 前后端测试通过
- 浏览器真实测试完成
- 代码检视完成
- 代码提交完成或已给出可执行提交方案
- 阻塞问题已显式记录
- 需回写文档已明确

## 18. 执行阻塞条件

出现以下任一情况，应暂停实施：

- 上游设计成熟度不足
- 共享契约不稳定
- 当前子任务边界不清
- 需要修改的文件显著超出预期
- 需求变更影响当前设计基础
- 当前存在未处理的 P0 / P1 阻塞问题

## 19. 需求变更时的暂停与回写规则

如果执行过程中出现需求补充、范围变化或设计冲突：

1. 不要直接继续改代码
2. 先判断变化类型：
   - 需求澄清
   - 范围变更
   - 验收标准变更
   - 技术约束补充
3. 再判断影响范围：
   - 当前子任务
   - 当前模块
   - 多模块 / 全局
4. 明确需要回写哪些文档
5. 如变化影响共享契约或模块边界，应暂停实施并上报总控 Codex
