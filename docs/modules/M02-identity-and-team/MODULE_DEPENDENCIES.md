## 2. M01 对 M02 的直接输入

### 2.1 技术与配置输入

- monorepo 结构、`apps/web` / `apps/api` / `infra`
- `.env` 读取基线
- FastAPI 路由挂载方式
- Next.js App Shell 与 i18n 入口约束

### 2.2 页面与测试输入

- `M01` 的工作台壳层决定 `/(dashboard)` 下受保护页面的承接方式。
- `M01` 的共享页面原语默认冻结口径已足以支撑 `/(dashboard)/members` 页面承接：`PageHeader` 只承载标题/说明/主次动作，摘要区独立承载状态与摘要项。
- `M01` 的列表原语与服务端分页思路影响 `/members` 的列表 envelope 设计。
- `M01` 的测试 / 日志基线要求 `M02` 从第一轮就建立鉴权矩阵测试文件。
- `M02` 已冻结 `/members` 为团队内基础成员目录，因此列表原语只需承接 `id`、`displayName`、`role`、`status` 四字段基线。
- 但 `M01` 当前仍未完全冻结：
  - 列表 query / URL / callback 的完整实现级接口细节
  - 完整的列表与页面实现仍不可绕开 `M01` 的共享 i18n 入口；`OQ-022` 已形成最小 locale namespace / fallback 默认口径，但 `M02` 不得私自扩张为完整 i18n 架构

## 5. 当前阻塞与风险

- 模块内部业务边界已基本收口；当前主要阻塞已转为共享契约，而非模块内业务歧义。
- 现有 `ST02_01 ~ ST02_03` 仍是遗留粗粒度任务包，其中 `ST02_01` / `ST02_02` 子任务设计还残留模板变量和超范围目标，不能继续作为正式入口。
- `MQ-205` / `OQ-021`
  - 影响：`MT02_04` / `MT02_06` 依赖 `/members` 列表的 envelope、分页参数、查询状态映射
  - 风险：若 `M02` 不按默认方案吸收，或继续私自定义列表协议，成员目录 API 与页面 adapter 仍会和 `M01` 共享列表原语冲突
- `MQ-206` / `OQ-022`
  - 影响：`MT02_05` / `MT02_06` 的页面文案入口、locale fallback 和 namespace 组织
  - 风险：若页面微任务先开，仍可能形成超出 `M01` 默认口径的模块私有 i18n 规则
- `MQ-203` / `OQ-023`
  - 影响：`MT02_07` 的职责边界与 admin 路由组 policy
  - 风险：若再次把完整成员管理实现回流到 `M02`，任务重切会重新失真
- `MQ-207`
  - 影响：下一轮是否沿用现有 `ST02_01 ~ ST02_03` 目录，还是按新的微任务边界重建子任务入口
  - 风险：若目录迁移策略不先说清，后续子任务 Codex 很容易再次沿用旧的粗粒度入口
- 模块内原有的命名 / 主键 / `logout` 语义漂移已在本轮收口，不再作为单独阻塞项。

## 6. 当前 readiness 判断

- `M02` 模块文档已具备“进入下一轮总控复核”的上游依赖。
- `M02` 当前不再适合继续用“模块内问题已清空”来推导 readiness；是否能进入子任务设计，应取决于共享契约阻塞是否解除、以及新的微任务入口是否被总控认可。
- 当前没有任何遗留 `ST02_01 ~ ST02_03` 适合直接开启子任务设计；真正最接近启动的是 `MT02_01`。
- 若要进入下一轮子任务设计，建议先完成：
  1. 由总控评审并确认新的微任务清单与遗留子任务目录迁移策略
  2. 由模块文档继续吸收 `MQ-205` / `OQ-021` 的默认冻结口径
  3. 由总控或相关模块完成 `MQ-206` / `OQ-022` 默认口径回写，并由 `M02` 吸收
  4. 由总控同步 `MQ-202`、`MQ-203`、`MQ-204` 已确认口径，并正式回写模块成熟度状态
