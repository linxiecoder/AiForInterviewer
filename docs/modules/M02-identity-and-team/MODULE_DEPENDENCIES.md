## 2. M01 对 M02 的直接输入

### 2.1 技术与配置输入

- monorepo 结构、`apps/web` / `apps/api` / `infra`
- `.env` 读取基线
- FastAPI 路由挂载方式
- Next.js App Shell 与 i18n 入口约束

### 2.2 页面与测试输入

- `M01` 的工作台壳层决定 `/(dashboard)` 下受保护页面的承接方式。
- `M01` 的共享页面原语默认冻结口径已足以支撑 `/(dashboard)/members` 页面承接：`PageHeader` 只承载标题/说明/主次动作，摘要区独立承载状态与摘要项。
- `M01` 的列表原语与服务端分页思路影响 `/members` 的列表 envelope 设计。
- `M01` 的测试 / 日志基线要求 `M02` 从第一轮就建立鉴权矩阵测试文件。
- `M02` 已冻结 `/members` 为团队内基础成员目录，因此列表原语只需承接 `id`、`displayName`、`role`、`status` 四字段基线。
- 但 `M01` 当前仍未完全冻结：
  - 列表 query / URL / callback 的完整实现级接口细节
  - 完整的列表与页面实现仍不可绕开 `M01` 的共享 i18n 入口；`OQ-022` 已形成最小 locale namespace / fallback 默认口径，但 `M02` 不得私自扩张为完整 i18n 架构

## 5. 当前阻塞与风险

- 模块内部业务边界已基本收口；当前主要阻塞已转为共享契约，而非模块内业务歧义。
- 现有 `ST02_01 ~ ST02_03` 仍是遗留粗粒度任务包，其中 `ST02_01` / `ST02_02` 子任务设计还残留模板变量和超范围目标，不能继续作为正式入口。
- `MQ-205` / `OQ-021`
  - 影响：`MT02_04` / `MT02_06` 依赖 `/members` 列表的 envelope、分页参数、查询状态映射
  - 风险：`M02` 已把 `/members` 共享列表契约闭合到共享最小层；若后续窗口又把 callback、URL 序列化或 request adapter 细节倒流回模块级协议，成员目录 API 与页面 adapter 仍会和 `M01` 共享列表原语冲突
  - 当前边界：这里的“已闭合”只解决模块内误补协议，不解决正式候选资格；因此既不能外推 `MT02_04` ready，也不能外推 `MT02_06` ready
- `MQ-206` / `OQ-022`
  - 影响：`MT02_05` / `MT02_06` 的页面文案入口、locale fallback 和 namespace 组织
  - 风险：若页面微任务先开，仍可能形成超出 `M01` 默认口径的模块私有 i18n 规则
- `MQ-207` / `OQ-024`
  - 影响：所有 `ST02_*` 与 `MT02_*` 的正式入口判定
  - 风险：全局虽已把“历史容器 / 观察蓝本 / 正式开窗层当前为空”写死；若模块文档不吸收该引用，后续窗口仍会沿用旧入口错误直开，或把观察蓝本误写成正式入口
- `MQ-209`
  - 影响：`MT02_01`、`MT02_02` 的 readiness 叙事，以及总控对白名单状态的回写
  - 风险：若不显式区分“白名单观察面”和“正式开窗状态”，或不把 `MT02_02` 的 `CurrentUserContext` / `401/403` / admin route group 消费边界继续锁在模块层，后续窗口会把本轮观察结论误写成可直接创建子任务窗口
- `MQ-203` / `OQ-023`
  - 影响：`MT02_07` 的职责边界与 admin 路由组 policy
  - 风险：若再次把完整成员管理实现回流到 `M02`，任务重切会重新失真
- 模块内原有的命名 / 主键 / `logout` 语义漂移已在本轮收口，不再作为单独阻塞项。

## 6. 当前 readiness 判断

- `M02` 模块文档已具备“进入下一轮总控复核”的上游依赖，但这不等于任何微任务已经获得子任务设计放行。
- `M02` 当前不再适合继续用“模块内问题已清空”来推导 readiness；当前最小剩余放行缺口已收敛为：
  1. `MODULE_API_DESIGN.md` 仍因 `GET /api/v1/members` 的共享最小层只停留在 `OQ-021` `proposed-default` 而维持高 `L4`
  2. 总控虽已写死 `OQ-024` 的三层映射，但正式开窗层当前仍为空，因此 `MT02_01 / MT02_02` 只能停留在观察蓝本
  3. `MT02_02` 的 `CurrentUserContext` / `logout=204` / `401/403` / admin route group 边界仍必须继续留在模块层，因为它们被 `MT02_05`、`MT02_07`、`MT02_08` 等共同消费，属于跨微任务共享契约，而不是单一 auth 子任务的私有输出
- 当前没有任何遗留 `ST02_01 ~ ST02_03` 适合直接开启子任务设计，也没有任何 `MT02_*` 已达到“可正式开窗”的状态。
- 当前轮次只能给出“白名单观察面”，而不是正式候选放行：
  - 白名单观察面：`MT02_01`、`MT02_02`
  - `MT02_01`、`MT02_02` 仍只是观察面：`MODULE_API_DESIGN.md` 仍是高 `L4` 最低位、`OQ-024` 正式开窗层当前为空，且遗留 `ST02_01` 仍是骨架 / 空白模板，不能作为正式入口证据
  - 仅保留后续观察顺序：`MT02_03`、`MT02_07`
  - 明确继续阻塞：`MT02_04`、`MT02_05`、`MT02_06`、`MT02_08`
  - `MT02_04` 的阻塞应读成“`/members` 已在模块内闭合到共享最小层，但仍只停留在 `OQ-021` 默认治理层 + API 最低位尚未跨过 `L5`”，而不是“接口快 ready”
  - `MT02_06` 继承 `MT02_04` 的全部阻塞，并额外受 `OQ-020 / OQ-022` 与页面 adapter 细节仍留在实现层的约束
- 白名单观察面不等于可开子任务窗口；它只表示总控可以继续跟踪这两个最小非页面变化面，而不是允许创建子任务设计窗口。
- 若要进入下一轮子任务设计，建议先完成：
  1. 由总控认可 `GET /api/v1/members` 当前共享最小层已足以作为正式候选输入，使 `MODULE_API_DESIGN.md` 不再因该引用维持高 `L4`
  2. 由总控在已写死的 `OQ-024` 映射基础上，正式确认哪些 `MT02_*` 可以从观察蓝本转入正式候选 / 开窗层
  3. 由总控同步全局文档，明确“白名单观察面”和“正式子任务候选”是两个状态，避免误开窗
  4. 继续把 `MT02_02` 的 `CurrentUserContext` / `logout=204` / `401/403` / admin route group 边界锁在模块层，不允许新的正式入口回流重定义
  5. 由总控或相关模块完成 `MQ-206` / `OQ-022` 默认口径回写，并由 `M02` 吸收，继续保持页面面后置
