# M02 鉴权、团队与成员 - Logic 设计

## 1. 文档定位

- 本文档用于沉淀 `M02` 的主流程、授权判断、异常路径、状态变化与验证要点。
- 当前目标是达到“可作为下游输入”的候选状态，待总控复核；因此仍显式保留未进入实施粒度的逻辑缺口，而不是硬写成实施定稿。

## 2. 逻辑输入基线

- `OQ-004`：P1 当前采用“JWT 前的开发态认证适配层”，对外保持 `Bearer token`
- `OQ-005`：权限矩阵首轮只覆盖 P1 页面与 API
- `Team` / `User` 为当前已冻结的持久化对象
- 所有团队作用域请求都必须通过 `CurrentUserContext` 提供 `team_id`

## 3. 主流程

### 3.1 登录流程

1. 客户端向 `POST /api/v1/auth/login` 提交账号密码。
2. `AuthResolver` 校验凭据并定位对应演示身份。
3. 若用户状态为 `active`，则返回开发态 `Bearer token` 与当前用户基础信息。
4. 前端保存登录态，后续请求统一携带 `Authorization` 头。

#### 关键规则

- 凭据和 token 都必须来自配置或种子数据。
- 同一身份重复登录应得到稳定、可复用的鉴权结果。
- 当前已冻结至少两组开发态身份映射：`admin` 与 `member` 各自拥有独立凭据与 token。
- `P1` 当前只允许 `active` 用户进入登录成功路径。
- 存储层到接口层的字段映射必须在序列化边界完成：
  - `display_name -> displayName`
  - `team_id -> teamId`

### 3.2 当前用户解析流程

1. 受保护请求进入鉴权依赖层。
2. 依赖层读取 `Authorization: Bearer <token>`。
3. 解析成功后产出 `CurrentUserContext`：
   - `user_id`
   - `team_id`
   - `role`
   - `status`
4. 业务服务只能消费该上下文，不接受客户端直接覆写团队归属。

#### 关键规则

- 缺失 token、token 无效、用户不可用时统一返回 `401`。
- 当前用户状态不是可登录态时，不应继续进入业务服务层。

### 3.3 成员目录列表 / 详情流程

1. 当前用户上下文解析成功。
2. 服务层按 `team_id + deleted_at IS NULL` 过滤成员。
3. 再按已冻结的成员目录规则裁剪字段与结果集。
4. 返回列表或详情安全投影。

#### 关键规则

- `/members` 已冻结为团队内基础成员目录，`admin` 与 `member` 都可读取本团队成员。
- 目录列表与详情都只返回：
  - `id`
  - `displayName`
  - `role`
  - `status`
- 无论最终可见范围如何，都不得把：
  - 跨团队成员
  - 软删除成员
  - 敏感字段
  返回给目录接口。
- 列表接口当前只冻结集合语义、条目投影与服务端分页方向。
  - 当前默认继承 `M01` 与 `OQ-021` 的最小共享映射：`page/page_size/q/status/sort/order` 与统一分页骨架
  - 在该共享契约进入 `confirmed` 前，`M02` 仍不应私自引入分页协议

### 3.4 管理台授权流程

1. 请求命中 `/(dashboard)/admin/**` 或 `/api/v1/admin/**`。
2. 当前用户上下文解析成功后，先判断 `role == admin`。
3. 角色不足时直接返回 `403`，不进入具体业务服务。
4. 角色满足后再进入后续模块实现的管理台逻辑。

#### 关键规则

- `M02` 负责这一判断基线。
- 已按 `MQ-203` 默认方案冻结：`M02` 只负责 admin 访问判定基线与最小鉴权契约，具体成员管理页面 / API 由后续治理模块承接。

### 3.5 退出登录流程

1. 客户端触发 `POST /api/v1/auth/logout`。
2. 后端确认当前登录态请求合法。
3. 后端返回 `204 No Content`。
4. 前端据此清空当前登录态；本轮不引入服务端 session 失效系统。

#### 关键规则

- 开发态认证适配层模式下，`logout` 应保持幂等。
- `logout` 成功返回 `204`，不依赖响应体承载额外字段。
- 若接入审计日志，日志失败不应让前端退出动作进入不一致状态。

## 4. 分支流程与异常路径

### 4.1 凭据错误

- 触发条件：邮箱不存在、密码不匹配、身份未启用。
- 返回：`401 invalid_credentials` 或统一 `401 unauthorized`。
- 处理原则：不泄露账户是否存在的细粒度信息。

### 4.2 未登录访问受保护资源

- 触发条件：缺少 `Authorization` 或 token 无效。
- 返回：`401 unauthorized`。
- 前端路径：重定向到登录页或展示统一登录失效态。

### 4.3 角色不足访问管理台

- 触发条件：`member` 访问 `/(dashboard)/admin/**` 或 `/api/v1/admin/**`。
- 返回：`403 forbidden`。
- 处理原则：应在授权层短路，而不是进入业务层后再失败。

### 4.4 跨团队成员详情访问

- 触发条件：当前用户访问非本团队详情资源。
- 返回：`403 forbidden`。
- 处理原则：成员目录路径统一表达“团队作用域拒绝访问”，不再额外区分不可见即不存在。

### 4.5 软删除对象访问

- 触发条件：访问已软删除成员或团队。
- 返回：
  - 详情 `404`
  - 列表默认过滤掉
- 处理原则：与计划中的软删除测试基线保持一致。

## 5. 状态变化

### 5.1 用户状态

- 当前评审基线只冻结：
  - `active` 用户可以登录与被目录返回
  - 软删除用户不可登录且不可见
- 其他用户状态值不进入 `P1` 当前模块契约。

### 5.2 团队状态

- 当前评审基线只冻结：
  - `active` 团队可被普通请求读取
  - 软删除团队不可见
- 其他团队状态值不进入 `P1` 当前模块契约。

### 5.3 登录相关状态

- 登录成功时，应允许更新 `last_login_at` 或至少保留该更新点。
- 开发态认证适配层模式下不维护复杂 session 生命周期。
- 退出登录不回写持久化 session 状态。

## 6. 并发 / 一致性 / 幂等性

- `POST /api/v1/auth/login`
  - 对同一身份重复调用应得到一致的身份结果。
  - 若 `last_login_at` 更新失败，不应把用户半登录到不一致状态。
- `GET /api/v1/auth/me`
  - 纯读取，天然幂等。
- `GET /api/v1/members*`
  - 纯读取，天然幂等。
  - 结果一致性依赖 `team_id` 与软删除过滤。
- `POST /api/v1/auth/logout`
  - 必须幂等。
  - 不因重复调用而引发异常状态。

## 7. 回滚 / 补偿原则

- 当前逻辑以读取与轻量登录态处理为主，不涉及跨资源分布式事务。
- 若登录附带 `last_login_at` 更新或审计写入失败：
  - 以“不破坏当前鉴权结果”为优先
  - 失败写入进入日志或补偿队列，而不是让前端处于未知登录态
- 若目录查询出现跨团队过滤缺失，应优先修复授权层 / 查询层，而不是在前端补遮罩。

## 8. 测试设计映射

- `auth` 基础测试：
  - 正确凭据 -> 成功返回 token 与 user payload
  - 错误凭据 -> `401`
- 鉴权矩阵测试：
  - 未登录 -> `401`
  - `member` 访问管理台 -> `403`
  - 本团队访问本团队资源 -> 成功
  - 跨团队成员目录详情 -> `403`
  - 软删除成员详情 -> `404`
  - 软删除成员不出现在列表

## 9. 当前逻辑缺口

- `logout` 的成功状态码已冻结为 `204`，不再作为当前轮阻塞项。
- 剩余逻辑阻塞主要来自共享契约：
  - `MQ-205` / `OQ-021`：`/members` 列表已具备统一默认映射，但仍需继续吸收到实现级页面 / callback 设计
  - `MQ-206` / `OQ-022`：登录页与成员页的 i18n namespace / fallback 已形成默认共享口径，但模块仍需完成吸收，不能私自扩张
